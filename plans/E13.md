# Plan — Epic E13: GM Console & player-view redesign (cockpit)

This epic replaces the original scrolling console with a single **cockpit**: a viewport-locked frame
whose edges always show the meters (Heat + Loot + phase/room), the crew, and the tool launchers,
with one calm work stage in the middle. It also reworks the mini-game lifecycle (arm → START →
resolve), adds a Spoils/Wrap-up beat, deletes the persistent gear tray, declutters the Getaway, and
redraws the player-view. Nothing in `main` for E13 is built yet (the empty `plans/E13.md` stub and
the "allow E13" orchestrator commit aside), so every task below is genuinely absent work. All the
*functional* surfaces it redraws already exist from E5–E12 — this is a **presentation/IA** epic.

**Hard constraint carried into every task (from the epic gate and `CLAUDE.md`):** do **not** change
engine rules, the `MiniGame` contract *shape*, content/presets, or design tokens. Three seams make
this possible without touching the engine: (a) the START gate is achieved by **deferring the game
component's mount until START** (its timer only runs once mounted — the contract is untouched);
(b) the Spoils beat is a **console-side presentation interstitial** between `RESOLVE_MINIGAME` (which
already sets `phase:'offer'`) and the Offer screen — no new engine phase; (c) all GM edits reuse the
existing `OVERRIDE_*` / `ASSIGN_GEAR` events. **Note for E13.7:** the engine drops **loot only** from
obstacles — gear is dropped solely by scenarios via `state.earnedGear`. The Spoils beat must announce
loot always and gear only when `earnedGear` is non-empty; it must **not** invent obstacle gear drops
(that would be an engine/content change → `PIPELINE_BLOCKED`). If any task hits a wall that genuinely
needs an engine/contract/content/token change, emit `PIPELINE_BLOCKED` rather than forking it.

Honour `docs/FRONTEND-REDESIGN-BRIEF.md` and the screen set in `design-system/redesign/`, and verify
tokens/type/voices/iconography against `design-system/colors_and_type.css`,
`design-system/README.md`, the two UI kits, and `design-system/preview/*`. The count is nine tasks:
the cockpit frame, the crew rail, the overlay system, the universal mini-game shell, two game-rework
batches (the ten games are independent and a single "rework all ten" task is too large for one
reviewer), the Spoils beat, the remaining phase-stage restage, and the Getaway + player-view. Each is
independently buildable, dependency-ordered, and reviewer-sized.

## Task E13.1 — Cockpit shell: viewport-locked frame, edge rails, action bar, overlay primitives
**Files:** `src/console/app.tsx` (replace the `.console` shell), new `src/console/shell/Cockpit.tsx`,
`src/console/shell/TopRail.tsx`, `src/console/shell/Stage.tsx`, `src/console/shell/ActionBar.tsx`,
`src/console/shell/overlays/{Drawer,Popover,Dialog}.tsx` + `index.ts`,
`src/console/shell/cockpit.css`; edits to `src/console/index.css` and `src/console/hud/Hud.tsx`
(top-rail meters move out of the old HUD); co-located tests.
**Do:** Build the fixed cockpit frame at a 1280×800 baseline that **never scrolls the document** —
only over-full *regions* scroll internally with an edge fade. Five stable regions: **top rail** =
meters (Heat track hero per `design-system/preview/comp-heat-track.html` + Loot + phase/room, always
visible); **left rail** = crew placeholder (filled in E13.2); **right rail** = tool-launcher slot +
persistent **Undo** (wired in E13.3); **bottom** = action bar with the phase's single primary CTA
(right), secondary/back (left), and a thin contextual-cue row (filled by E13.7/soundboard); **centre**
= the work stage with a **fixed-height teleprompter strip** at its top (one beat, `t-teleprompter`,
`Next` — never grows/pushes content). Provide reusable **overlay primitives**: `Drawer` (slides from
an edge, scrim, stage stays visible), `Popover` (anchored to a trigger), `Dialog` (centred, blocking,
scrim) — each with Esc-to-dismiss and focus handling. Existing phase screens render unchanged inside
the stage for now (restyled in E13.5/6/7/8/9). Keep `data-testid` anchors the existing tests rely on
(`hud`, `hud-heat-section`, `hud-loot-section`, `loot-total`, `screen-*`) or update those tests.
**Acceptance:** at 1280×800 the document/`body` never scrolls (assert via the shell test that the
scroll container is an inner region, not the page); top rail shows Heat + Loot + phase + room in every
phase; teleprompter is a fixed-height strip; the three overlay primitives mount and dismiss via
scrim/Esc; tokens/voices verify against `design-system/preview/*`; `npm run check:full` passes.
**Depends:** none (builds on merged E5–E12).

controls — ensure **no per-player editing capability is lost** in the move.
**Acceptance:** crew is visible in every phase; the popover dispatches only existing `OVERRIDE_*`

## Task E13.4 — Universal mini-game shell: ARMED → START → ACTIVE → RESOLVE, standard zones
**Files:** rewrite `src/console/screens/MinigameHost.tsx`; new
`src/minigames/primitives/MinigameShell.tsx` and zone helpers
(`StatusZone`/`ChallengeZone`/`RefereeZone`); reuse `DialReadout`/`OutcomeJudge`/`BoostButton`;
`src/minigames/primitives/minigame.css`; update `MinigameHost.*.test.tsx`.
**Do:** Wrap **every** game in one shared three-state lifecycle. **ARMED** (default on entry): show the
game name, the GM read-aloud script (from the narration director), the GM-only `DialReadout`, the
boosts-available preview (which committed holder holds which lane boost), and one big **`START`** CTA
in the action bar — and **do not mount the game component**, so no timer can run on load. **ACTIVE**
(after START): mount the game inside standard stage zones — **status** (mode · timer · progress) ·
**challenge** (the hero) · **referee** (boosts · outcome). **RESOLVE**: when the game calls
`onResolve`, present the outcome control (Clean/Complication/Botched) with the suggested tier
pre-selected and the comedic middle one tap away; the GM **confirms** (the app never overrides) →
hand off to the Spoils beat (E13.7; until then keep the existing `RESOLVE_MINIGAME` dispatch and
outcome-quip path). The START gate is the deferred-mount described in the plan preamble — the
`MiniGame` contract shape is unchanged. Per-game visual rework (visible state, meter progress,
no-layout-shift, card affordances, static ARMED preview) is E13.5/E13.6.
**Acceptance:** for **all ten** games the clock does not run until START (assert no running `Timer`
in the ARMED state across the host tests); standard status/challenge/referee zones are present in
ACTIVE; RESOLVE pre-selects the suggested outcome and only the GM's confirm dispatches; the
unregistered-game fallback still never dead-ends; `npm run check:full` passes.
**Depends:** E13.1.
