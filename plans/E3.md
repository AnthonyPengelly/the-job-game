# Plan — Epic E3: Room loop & GM console shell (mini-games stubbed)

**Approach.** The pure engine, crew/gear/scaling, and GM-override events all shipped in E1/E2 (`reduce`, `reduceSession` + `UNDO_LAST`, the `OverrideEvent` union, `ASSIGN_GEAR`, `RESOLVE_MINIGAME`, `RESOLVE_GETAWAY`). E3 adds **no engine code** — it is the React console that drives that engine from the UI, plus the store boundary and refresh-safe persistence that the architecture says live here. The work is: (1) a browser-capable config loader (the existing `loadPreset` uses `node:fs` and can't run in the browser), (2) the persisted-save Zod schema + a `localStorage` adapter, (3) the Zustand store that wraps `reduceSession`, records the event log, and write-throughs/hydrates, then (4–10) the React screens, HUD, gear direct-manipulation, override surface, and the mini-game stub picker.

Persistence uses the architecture's `{ seed, eventLog }` model: the store records every dispatched `RunEvent` (dropping the last on `UNDO_LAST`), write-throughs after every event, and on boot replays the log from `initialState(seed)` to reconstruct the exact `SessionState`. Saves are schema-versioned and Zod-parsed; a stale/mismatched/corrupt save fails loudly and offers a clean restart, never a corrupted run. Everything tunable stays preset data; no new randomness is introduced; the player-view surface is untouched (it remains an E5/E6 concern). Every override goes through `reduce`, and `OVERRIDE_SET_PHASE` + `UNDO_LAST` guarantee no reachable state is a dead-end.

A note on the "drag a dropped loot/gear card onto the player" story: in E3 there is no room mechanism that *drops* gear (that is E7 content) — the assignable gear is the preset gear catalog (`cfg.gear`). Direct-manipulation therefore assigns catalog gear to a player via the existing `ASSIGN_GEAR` event; gear *removal*/adjustment is via the stat/power-up overrides on the GM surface. No new engine event is required, and the epic's "grant/remove any gear" capability is fully covered.

Component tests run under jsdom via a per-file `// @vitest-environment jsdom` docblock (vitest's default env stays `node` for the engine); store/persistence logic is tested in plain node with an injected in-memory storage and `test-config`. The full playable-run, HUD-correctness, override/undo, no-dead-end, and refresh-resume assertions are the epic's QA-agent gate (Playwright).

## Task E3.3 — Zustand store: engine wrapper + write-through persistence + hydrate/replay
**Files:** `package.json` (+`zustand`), `src/console/store/store.ts` (new), `src/console/store/replay.ts` (new), `src/console/store/store.test.ts` (new), `src/console/store/index.ts` (new)
**Do:** Add `zustand`. The store holds the current `SessionState` (from `@/engine`'s `initialSession`/`reduceSession`), the recorded `eventLog: RunEvent[]`, and the boot-loaded `EngineConfig` (injected; defaults to `loadDefaultConfig()`). Actions: `dispatch(event: RunEvent)` → `reduceSession` (which pushes undo history) **and** append the event to `eventLog`, then write-through `{ version, seed, eventLog }` after the state settles; `undo()` → dispatch `UNDO_LAST` to `reduceSession` **and** drop the last entry from `eventLog`, then write-through; `startRun(setup, seed)` → reset session + log to a single `START_RUN`; `goAgain()` → `clearSave()` and reset to a fresh briefing/setup. `replay.ts` exports a pure `replay(seed, eventLog, cfg): SessionState` that folds `reduce` from `initialState(seed)`. On store creation, `hydrate()` calls `readSave()`; on `ok` it `replay`s into the store and exposes `hasResumableSave = true`; on `stale`/`corrupt` it `clearSave()`, surfaces a loud one-time notice flag, and starts clean. Store is framework-thin: it never reimplements a rule, only calls `reduceSession`/`reduce`.
**Acceptance:** `store.test.ts` (node, injected in-memory storage + `test-config`): a sequence of dispatches advances state via the engine and write-throughs after each; `undo()` restores prior state **and** shrinks `eventLog`; a fresh store hydrating from that storage `replay`s to a byte-identical `present` (proves `{seed,eventLog}` resume); a `stale` save → clean start + notice flag + cleared storage. `npm run check:full` green.
**Depends:** E3.1, E3.2
