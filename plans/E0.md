# Plan — Epic E0: Foundations & scaffold

**Approach.** E0 stands up a runnable, empty React + Vite + TypeScript app and the
full toolchain the rest of the pipeline leans on: strict TS with path aliases, the
six-layer `src/` skeleton, Vitest, the seeded RNG, the import-direction lint rule
(with a sensor that proves it bites), a design-token skeleton, and the real
`check` / `check:full` gates. Nothing here is application logic — it is the floor
every later epic builds on, so the bar is "green, deterministic, and the
boundaries are mechanically enforced," not "does anything yet."

Tasks are ordered so each depends only on earlier ones. E0.1 lays the toolchain
and folder skeleton; E0.2 adds the test runner; E0.3 adds the one piece of real
engine code E0 ships (the RNG) so determinism is provable from day one; E0.4 wires
the lint boundary and its sensor; E0.5 adds the CSS-variable theme placeholder;
E0.6 fuses everything into the `check` / `check:full` gates that are the epic's
acceptance contract. Honour the golden rules throughout: the engine stays pure
(no React/DOM/timers/`Math.random`), the import-direction rule is the mechanical
guard on the one-way dependency flow, and the design-token work is a *skeleton*
only — the cinematic visual pass is explicitly E10 (`the-job-app-design.md` §"Design
tokens / vibe … scheduled late").

## Task E0.1 — Scaffold the Vite + React + TS toolchain and six-layer skeleton
**Files:** `package.json`, `tsconfig.json`, `tsconfig.node.json`, `vite.config.ts`, `index.html`, `src/main.tsx`, `src/vite-env.d.ts`, `src/console/app.tsx`, `src/console/index.css`, `src/{engine,content,minigames,console,player-view,platform}/index.ts`
**Do:** Initialise a Vite + React 18 + TypeScript SPA. Strict `tsconfig` (`strict`, `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`, `noUnusedLocals/Parameters`, `noFallthroughCasesInSwitch`, `verbatimModuleSyntax`) with the `@/engine`, `@/content`, `@/minigames`, `@/console`, `@/player-view`, `@/platform` path aliases. Create the six-layer `src/` skeleton (`engine`, `content`, `minigames`, `console`, `player-view`, `platform`), each with a placeholder `index.ts` documenting the layer; render a minimal GM-shell from `src/console/app.tsx` via `src/main.tsx`. Wire `dev`, `build`, `typecheck`, `lint`, `test` npm scripts (gates `check`/`check:full`/`sim:check`/`content:validate` may remain labelled stubs until their owning epic). No application logic.
**Acceptance:** `npm run dev` serves a shell at localhost; `npm run build` and `npm run typecheck` pass; aliases resolve in both `tsconfig` and Vite; the six `src/*` folders exist with no upward imports.
**Depends:** none

## Task E0.2 — Configure Vitest with one passing example test
**Files:** `vite.config.ts` (or `vitest.config.ts`), `src/example.test.ts`, `package.json` (`test` script)
**Do:** Configure Vitest, sharing the Vite alias resolution so tests use the same `@/…` paths as the app (consolidate alias config in one place — do not maintain a second divergent alias map). Add one trivial passing test so `npm run test` is green and the runner is proven wired.
**Acceptance:** `npm run test` runs Vitest and passes; the example test resolves at least one `@/…` alias to prove alias parity between app and tests.
**Depends:** E0.1

## Task E0.3 — Seeded RNG in the engine with reproducibility tests
**Files:** `src/engine/rng.ts`, `src/engine/rng.test.ts`
**Do:** Implement a seedable `mulberry32` PRNG in `src/engine/rng.ts` as the engine's single randomness source — a factory taking a numeric seed and returning a deterministic generator (e.g. `next()` in `[0,1)`, plus minimal helpers like an integer-in-range if needed). Pure: no `Math.random`, no `Date.now`, no module-level mutable singleton; the instance is created from an explicit seed and threaded by callers. Co-locate tests proving same seed ⇒ identical sequence, and different seeds ⇒ different sequences.
**Acceptance:** `npm run test` green; the reproducibility test asserts `mulberry32(s)` produces an identical sequence to a second `mulberry32(s)` and a different one for a different seed; nothing engine-side references `Math.random`/`Date.now`.
**Depends:** E0.1, E0.2

## Task E0.4 — ESLint + Prettier and the import-direction rule with a proving sensor
**Files:** `eslint.config.js`, `.prettierrc`, `.prettierignore`, `scripts/sensors/lint-import-direction.mjs`, `package.json` (`lint` script)
**Do:** Configure ESLint (flat config, TS-aware) + Prettier (via `eslint-config-prettier`). Encode the one-way dependency flow from `docs/ARCHITECTURE.md` §1/§3 as an import-direction rule (`no-restricted-imports` zones): `src/engine/**` may not import React/DOM/`@/content`/`@/minigames`/`@/console`/`@/player-view`/`@/platform` and may not use `Math.random`; `src/content/**` → `@/engine` only; `src/minigames/**` → engine/content/platform, never the surfaces or another game's internals; `src/console/**` not `@/player-view`; `src/player-view/**` not `@/console`; `src/platform/**` not surfaces/minigames. Cover both alias (`@/…`) and relative-path forms. Add `scripts/sensors/lint-import-direction.mjs`: it writes a throwaway file that imports React into the engine, runs ESLint, asserts ESLint **fails** (rule fired → sensor exits 0), and asserts a sensor failure if ESLint passes; the temp file is always cleaned up.
**Acceptance:** `npm run lint` passes on the clean tree; `node scripts/sensors/lint-import-direction.mjs` exits 0 (it proves an engine→React import is rejected and self-cleans), and exits non-zero if the rule ever stops firing — this is the epic gate's "prove the rule bites, delete after" requirement, automated.
**Depends:** E0.1

## Task E0.5 — Design-token skeleton (CSS variables; dark "van" theme placeholder)
**Files:** `src/console/theme/tokens.css` (+ `src/console/theme/index.ts` if a typed token re-export is useful), `src/console/index.css` (import the tokens), `src/main.tsx` (ensure the stylesheet loads)
**Do:** Add a CSS-variables skeleton for the dark, cinematic "van" aesthetic as a **placeholder only** — define a small, named set of design tokens on `:root` (e.g. background/surface, foreground/muted text, an accent, a danger/Heat colour, a couple of spacing/radius/font tokens) with dark default values, and have the console baseline consume a few of them so they are demonstrably live. Do **not** attempt the full visual pass — `the-job-app-design.md` schedules cinematic polish for E10; this is scaffolding so later epics theme against named variables instead of hardcoded hex.
**Acceptance:** `npm run dev` shows the shell rendering against the dark token values (at minimum background + foreground come from CSS variables); `npm run build`/`typecheck`/`lint` stay green; tokens live in `src/console/theme/` and are referenced (not redefined) by `src/console/index.css`.
**Depends:** E0.1

## Task E0.6 — Wire the real `check` and `check:full` gates
**Files:** `package.json` (`check`, `check:full` scripts), `scripts/sensors/` (only if a sensor runner aggregator is added)
**Do:** Replace the `check` and `check:full` stub scripts with real composite gates. `check` = fast inner loop: `typecheck` + `lint` + `test`. `check:full` = `typecheck` + `lint` + `test` + sensors, where sensors include `node scripts/sensors/lint-import-direction.mjs`. Leave `sim:check` and `content:validate` as their documented stubs (owned by E1 and E7/E10 respectively) so the pipeline still bootstraps; update the `comment:gates` note in `package.json` to reflect that E0.6 has now wired `check`/`check:full`. Fail-fast composition (a failing sub-step fails the gate).
**Acceptance:** This closes the **epic acceptance gate**: `npm run check:full` passes on the empty app (typecheck + lint + tests + the import-direction sensor all green); `npm run dev` serves a shell; the RNG reproducibility test is green; and the import-direction sensor proves the lint rule fails on an engine→React import. `npm run check` runs the fast subset and passes.
**Depends:** E0.1, E0.2, E0.3, E0.4, E0.5
