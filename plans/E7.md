# Plan — Epic E7: Scenario content & rolls

**Approach.** Today the engine has only a placeholder scenario model: six minimal templates inside `roomTemplates.json`, each choice carrying a flat `heatDelta`/`lootDelta`, and `CHOOSE_SCENARIO` simply applies that swing. There are no lane-weighted rolls, no d20 DC, no Gear/info/delayed currencies, no two-stage reveal, and no dice-mode setting. E7 replaces that placeholder with the real thing: all 44 scenarios from `docs/design/heist-content.md` as validated data, a seeded (or physical) d20 roll resolution with the transparent `DC = baseDifficulty − laneRating` reveal, and resolution of *every* currency the scenarios pay out (Heat / Loot / Gear / info / delayed).

The work splits cleanly along the dependency flow. **E7.1** builds the pure engine model — the effect/roll types, `DC`/roll/crit-fumble math, the `RESOLVE_SCENARIO_ROLL` event, and resolution of all five currencies (Gear via an earned-gear pool the GM assigns through the existing `GearTray`; info/delayed via carried effects) — tested against inline fixtures so it needs no real content. **E7.2** authors the 44 scenarios as a new validated content pack against that model and wires the loader/generation to draw them. **E7.3** adds the persisted dice-mode setting (its own platform/persistence key + a console settings control). **E7.4** rebuilds the `ScenarioRoom` screen into the blind→transparent two-stage reveal that consumes the engine state and the dice-mode setting. Determinism, GM-override/no-dead-ends, parse-at-the-boundary, and the pure-engine line are honoured throughout: the entered physical roll is recorded in the `RESOLVE_SCENARIO_ROLL` event so `{seed, eventLog}` replays exactly and a mistyped roll is `UNDO_LAST`-able.

## Task E7.1 — Engine: scenario roll resolution & all five currencies
**Files:** `src/engine/types.ts`, `src/engine/scenario.ts` (new), `src/engine/reduce.ts`, `src/engine/generation.ts`, `src/engine/config.ts`, `src/content/schema/tuning.ts`, `src/content/schema/save.ts`, `presets/default/tuning.json`, co-located `*.test.ts`.
**Do:**
- **Types.** In `src/engine/types.ts` add the canonical scenario model the engine resolves (content will validate JSON into these): `ScenarioEffect` — a structured outcome carrying `{ heatDelta, lootDelta, gear?, info?, delayed? }` where `gear` is a grant descriptor (`{ kind: 'statBoost'|'powerUp'|'bigScore', lane: Lane }` or a `lanes: Lane[]` choice when the scenario says "lane of choice" / "Charm or Stealth"), `info` flags ease-the-next-room, and `delayed` describes a carried payoff (`{ kind, roomsLeft, payoff: ScenarioEffect }`, e.g. the ticking briefcase). `ScenarioRollSpec` — `{ lane: Lane, baseDifficulty: number, success: ScenarioEffect, failure: ScenarioEffect }`. A scenario `ScenarioChoiceDef` is either `{ effect: ScenarioEffect }` (no-roll) or `{ roll: ScenarioRollSpec }`. `ScenarioDef` — `{ id, setup, choices: [ScenarioChoiceDef, ScenarioChoiceDef] }` (each choice keeps its `id` + flavour `label`). Add a `PendingRoll` shape on `RunState`/`ScenarioRoom` capturing the *revealed* values: `{ choiceId, attemptedBy, lane, laneRating, baseDifficulty, dc }`. Add an `earnedGear: GearId[]` pool to `RunState` (scenario Gear rewards land here for the GM to assign). Add the new event `{ t: 'RESOLVE_SCENARIO_ROLL'; externalRoll?: number }` to `RunEvent`. Extend `RoomResult`'s scenario variant to record the roll (`roll?`, `dc?`, `success?`).
- **`src/engine/scenario.ts` (new, pure).** `computeDC(baseDifficulty, laneRating, clamp)` → clamp `baseDifficulty − laneRating` to `[1,20]`; `successOdds(dc)` → `(21 − dc) / 20`; `resolveRoll(roll, dc, critFumble)` → success on `roll ≥ dc`, with optional nat-20 crit / nat-1 fumble override when the preset flag is on; `applyScenarioEffect(state, effect, cfg)` → resolves *every* currency: Heat (via the existing `applyScenarioSwing` clamp), Loot, Gear (push the resolved `GearId` from `cfg.gear` matching the descriptor onto `earnedGear`), info (spawn an `easeNextObstacle` carried effect), delayed (spawn the described carried effect). Keep it deterministic and side-effect-free.
- **Carried-effect firing.** Extend `tickCarriedEffects` (in `generation.ts`) so an effect that reaches expiry **fires its payoff** rather than silently dropping (briefcase → `Loot++`); and so an active `easeNextObstacle` effect lowers the next obstacle's dial by `cfg.scenario.easeDialSteps` (default 1) at generation/commit. Thread the fired payoffs back through `applyScenarioEffect`.
- **Reducer.** `CHOOSE_SCENARIO`: for a **no-roll** choice apply its `effect` and advance to `offer` (current behaviour, richer effect); for a **roll** choice require `attemptedBy`, compute and store `pendingRoll` (lane, attempter's `laneRating`, `baseDifficulty`, `dc`) and **stay in the room** awaiting the roll — do not advance. New `RESOLVE_SCENARIO_ROLL`: read `pendingRoll`; use `event.externalRoll` when present, else draw a d20 from the seeded RNG; resolve success/failure, apply the chosen `ScenarioEffect`, record the roll in history, clear `pendingRoll`, advance to `offer`. Keep `default: never` exhaustiveness.
- **Config/tuning.** Add `cfg.scenario` (`dcClamp: [1,20]`, `easeDialSteps`, `critFumble: boolean` default `false`) to `EngineConfig` + the `tuning` Zod schema + `presets/default/tuning.json`. Replace the minimal `ScenarioTemplateConfig`/`ScenarioChoiceConfig` in `config.ts` with the new `ScenarioDef` shape consumed by generation.
- **Save schema.** Add `RESOLVE_SCENARIO_ROLL` (with optional integer `externalRoll`) to `src/content/schema/save.ts` so the event log validates and replays; `CHOOSE_SCENARIO.attemptedBy` already exists there.
- **Generation.** Project the drawn `ScenarioDef` into `currentRoom`: carry `setup` + each choice's flavour `label` + whether each choice is a roll (so the UI can stay opaque pre-commit). No-repeat draw already exists — keep it.
**Acceptance:** unit tests cover: DC computation and the `[1,20]` clamp; `successOdds`; app-roll resolution is seeded & reproducible (same seed+events ⇒ same roll); `externalRoll` is used verbatim when present; each currency resolves (Heat, Loot, Gear→`earnedGear`, info→ease carried effect, delayed→fires payoff on expiry, e.g. briefcase Loot++); crit/fumble flag on/off; `UNDO_LAST` restores the prior state after a roll (mistyped physical roll is reversible). `npm run check:full` green (typecheck, lint, tests, `sim:check`, `content:validate`).
**Depends:** none (E2/E3 merged).

## Task E7.2 — Content: scenario schema + all 44 scenarios + loader wiring
**Files:** `src/content/schema/scenarios.ts` (new), `src/content/schema/index.ts`, `src/content/index.ts`, `presets/default/content/scenarios.json` (new), `src/content/schema/room-templates.ts` (drop the superseded minimal scenario schema), `src/engine/config.ts`/loader wiring, co-located `*.test.ts`.
**Do:**
- Write a Zod schema (`scenariosSchema`) in `src/content/schema/scenarios.ts` that validates the new content pack into the engine's `ScenarioDef` model from E7.1: pack `_meta`, then an array of scenarios with `id`, `setup`, and exactly two choices, each choice being either `{ effect }` or `{ roll }`, with strict objects, `lane ∈ {tech,physical,charm,stealth}`, integer `baseDifficulty`, and Gear descriptors referencing valid catalogue gear kinds/lanes. Parse at the boundary; malformed content fails loudly.
- Author **all 44 scenarios** from `docs/design/heist-content.md` into `presets/default/content/scenarios.json`, faithfully mapping each option to currencies: Heat ↑/↓/⇈/⇊ → small/big swings, Loot +/++, **Roll (lane)** options → `roll` specs with the named lane, `+stat boost`/`+power-up`/Big Score → Gear descriptors, *info* → `info`, and the delayed payoffs (Locked Briefcase 3-room→Loot++, Blueprints next-room-easier, "they owe you"). Calibrate `baseDifficulty` so a 0-rating attempt is roughly a coin-flip and a boosted specialist visibly lowers the DC (target band ≈ 11–16; never a guaranteed pass), consistent with the lane-rating scale (stats start at 0, +1 boosts stack). No tunable numbers in code — all of this is preset data.
- Wire the loader/`EngineConfig` so generation draws scenario rooms from this pack (replace the old `roomTemplates.scenarios`); keep obstacles in the room-templates pack. Update `src/content/index.ts`/`schema/index.ts` exports.
**Acceptance:** validation test asserts all **44** scenarios parse against the schema, the count is exactly 44, every roll choice has a valid `lane` + integer `baseDifficulty`, and every Gear descriptor resolves to a real catalogue gear. A run draws scenarios with **no repeat within the run** (generation test over a long run). `npm run content:validate` and `npm run check:full` green.
**Depends:** E7.1.

## Task E7.4 — Console: two-stage scenario roll reveal UI
**Files:** `src/console/screens/ScenarioRoom.tsx`, any small supporting component(s), `src/console/screens/ScenarioRoom.test.tsx`.
**Do:** Rebuild `ScenarioRoom` into the blind→transparent flow (`docs/design` §10.3):
- **Stage one (opaque):** show the scenario `setup` and the two choice flavour `label`s only — **no lane, no DC, no Heat/Loot numbers**.
- **On selecting a roll choice:** prompt the GM to pick *which crew member attempts* it, then dispatch `CHOOSE_SCENARIO { choiceId, attemptedBy }`. The engine returns a `pendingRoll`.
- **Stage two (transparent):** reveal the lane, the attempter's rating in it, the base difficulty, the **d20 DC** (`DC = baseDifficulty − laneRating`), and the odds `(21 − DC)/20`. Then resolve the roll per the persisted dice-mode: **app** → a single "Roll" button dispatching `RESOLVE_SCENARIO_ROLL {}` (engine draws the seeded d20); **physical** → a 1–20 input that validates the entry and dispatches `RESOLVE_SCENARIO_ROLL { externalRoll }`. Show the resolved success/failure and the currencies paid out (including any earned Gear surfaced for assignment via the existing `GearTray`/`ASSIGN_GEAR`, and info/delayed effects).
- **No-roll choices:** select → reveal the effect → confirm → `CHOOSE_SCENARIO { choiceId }` advances to Offer.
- Scenarios are GM-only — render nothing to the player-view. Every step remains GM-overridable and `UNDO_LAST`-able (don't bypass the store/reducer).
**Acceptance:** component tests prove: stage one shows flavour only (no lane/DC/odds leaked pre-commit); after commit + attempter pick, the revealed DC equals `clamp(baseDifficulty − laneRating, 1, 20)` and odds read `(21−DC)/20`; **app mode** resolves via the seeded RNG and is reproducible; **physical mode** accepts a 1–20 entry, rejects out-of-range, and the resulting roll is `UNDO_LAST`-able; switching dice mode (E7.3) changes which control appears; no-roll choices resolve directly. `npm run check:full` green.
**Depends:** E7.1, E7.3 (and reads E7.2 content at runtime; functions against fixtures for tests).
