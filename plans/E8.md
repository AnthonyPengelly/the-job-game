# Plan — Epic E8: Narration bank & teleprompter

**Approach.** Nothing for E8 exists on `main` yet: there is no narration schema, no
narration content pack (the `_meta.json` lists `content/narration.json` but no file or
loader exists), no variant selector, and no teleprompter — the run screens (`Briefing`,
`ObstacleRoom`, `ScenarioRoom`, `Offer`, `MinigameHost`, `Getaway`, `Result`) currently
hardcode their flavour strings. So this whole epic is net-new.

We build it bottom-up along the dependency flow `content → console`, keeping the engine
untouched (narration is presentation/flavour, never balance — it must not enter
`EngineConfig` or perturb the engine RNG stream, or the sim/balance numbers would shift).
First the **content layer**: a Zod schema for a per-beat variant bank plus a *pure*
non-repeating picker (E8.1), then the authored default bank and its preset-loader wiring
(E8.2). Then the **console layer**: a `NarrationDirector` that owns a seeded RNG derived
from the run seed (a separate, salted `mulberry32` stream so it never touches engine
state) and the per-beat recently-used buffers, plus the teleprompter presentation
component (E8.3). Finally we replace the hardcoded strings screen-by-screen, in two
review-sized passes (E8.4 pre-Offer beats, E8.5 outcome + endgame beats).

Determinism: same seed ⇒ same narration sequence, because the director seeds its picker
from `runSeed` and advances deterministically as beats are requested in run order. The
"no repeated line across a full run" gate is proven headlessly in E8.3 by walking a full
run's beat sequence through the director and asserting no repeats. Golden rules honoured:
the bank is data in the preset (parsed at the boundary with Zod); the director lives in
the console and never imports upward; narration only ever *adds* lines to the GM's screen
and can never block an action, so GM-override / no-dead-ends is preserved untouched.

## Task E8.1 — Narration schema & pure non-repeating selector
**Files:** `src/content/schema/narration.ts` (+ `narration.test.ts`), `src/content/narration/select.ts` (+ `select.test.ts`), `src/content/narration/index.ts`, exports added to `src/content/schema/index.ts` and `src/content/index.ts`
**Do:**
- Define `narrationSchema` (Zod): a bank keyed by **beat** for every beat the epic names —
  `briefing`, `obstacleClue`, `optionDescription`, `pushRun`, `outcomeQuip`, `scenarioSetup`,
  `getawayIntro`, `getawayCountdown`, `winSting`, `bustSting`. Each beat holds an array of
  **variants**; a variant is `{ id, text, when? }` where the optional `when` is a typed
  condition object used to scope a variant to context — supported keys: `mansionType`
  (`'villa' | 'estate' | 'penthouse'`), `gameId` (one of the ten `GameId`s),
  `lane` (`'tech' | 'physical' | 'charm' | 'stealth'`), `outcome`
  (`'clean' | 'complication' | 'botched'`), `greedy` (`boolean`), `heatBand`
  (`'cool' | 'warm' | 'hot'`). Import these unions as **types** from `@/engine` only (no
  value imports — content may import engine types only). Variant `id`s must be unique within
  a beat (schema `superRefine`).
- Export `ParsedNarration` and a `NarrationBeat` union type.
- Implement `selectVariant(variants, recentIds, rand)` — a **pure** picker: given the
  already-context-filtered candidate variants, the recently-used id ring buffer for that
  beat, and a `rand: () => number` in `[0,1)`, return the chosen variant. It prefers
  candidates **not** in `recentIds`; when all candidates are recent (small pool exhausted)
  it falls back to the full candidate set so it never throws. Deterministic given the same
  `rand` sequence. No engine/RNG/React import — `rand` is injected.
- Also export a small pure `filterByContext(variants, ctx)` helper that keeps variants whose
  `when` (if present) matches the supplied partial context.
**Acceptance:**
- `narrationSchema` parses a valid fixture and rejects (a) a beat with duplicate variant ids
  and (b) an unknown `when` key — both asserted in `narration.test.ts`.
- `select.test.ts` proves: deterministic selection under a fixed `rand`; never returns a
  recently-used id while a fresh candidate exists; falls back without throwing when the pool
  is exhausted; `filterByContext` scopes correctly (e.g. a `mansionType:'villa'` variant is
  excluded for an `estate` context, unconditioned variants always pass).
- `npm run check:full` green (typecheck, lint incl. import-direction, tests).
**Depends:** none (E0–E7 merged)

## Task E8.2 — Author the default narration bank & wire the preset loader
**Files:** `presets/default/content/narration.json`; `src/platform/presets/load.ts` and `src/platform/presets/browser.ts` (add `loadNarration` / `loadDefaultNarration`); `src/platform/presets/load.test.ts` / `browser.test.ts`; `src/content/narration/counts.test.ts` (or extend an existing content test); update `package.json` `content:validate` to parse `narration.json` against `narrationSchema`
**Do:**
- Author `narration.json` conforming to `narrationSchema`, with **many** variants per beat so
  a full run never repeats. Author tone consistent with the "guy in the van" voice and the
  mansion dressing (no banks/museums — villa/estate/penthouse only). Minimum coverage,
  weighted to how often each beat fires in a run:
  - `briefing`: ≥3 per `mansionType` (≥9 total).
  - `obstacleClue`: ≥4 per `gameId` (covering all ten games) — high frequency, needs depth.
  - `optionDescription`: ≥4 for greedy and ≥4 for safe (optionally lane-flavoured).
  - `scenarioSetup`: ≥6 framing lines (intro to "the van reads you a situation"; the scenario
    body text itself stays authored per-scenario in `scenarios.json` — these are the wrapper).
  - `outcomeQuip`: ≥6 each for `clean`, `complication`, `botched` (fires every obstacle).
  - `pushRun`: ≥4, with at least some `heatBand:'hot'` variants for the escape-signal moment.
  - `getawayIntro`: ≥4; `getawayCountdown`: ≥4.
  - `winSting`: ≥4; `bustSting`: ≥4.
- Add a `loadNarration(dir)` to the Node loader and `loadDefaultNarration()` to the browser
  loader that read + `narrationSchema.parse` the pack and return `ParsedNarration`. **Do not**
  fold narration into `EngineConfig`/`buildConfig` — it returns separately so the engine and
  sim stay narration-free.
- Wire `content:validate` so the narration pack is schema-validated in CI (replace/extend the
  current stub so it fails loudly on a malformed pack).
**Acceptance:**
- `narration.json` parses; loader tests assert `loadDefaultNarration()` returns a parsed bank
  and that a deliberately malformed fixture throws.
- A counts test asserts the per-beat minimums above (so the bank can't silently regress below
  "enough variants for a no-repeat run").
- `npm run content:validate` passes and now actually validates narration; `npm run check:full`
  green.
**Depends:** E8.1

## Task E8.3 — NarrationDirector & teleprompter presentation
**Files:** `src/console/teleprompter/director.ts` (+ `director.test.ts`), `src/console/teleprompter/Teleprompter.tsx` (+ `Teleprompter.test.tsx`), `src/console/teleprompter/index.ts`; minimal store wiring in `src/console/store/store.ts` (hold/reset a director seeded from `runSeed`) with a test in `src/console/store/store.test.ts`
**Do:**
- Implement `createNarrationDirector(bank, seed)`: owns its own `mulberry32` seeded from the
  run seed **salted** with a fixed constant (so it is reproducible from the seed yet never
  shares or advances the engine's RNG stream) and a per-beat recently-used ring buffer
  (configurable window, default ~ half the beat's pool). Expose `next(beat, ctx?)` which
  filters the beat's variants by `ctx` (`filterByContext`), calls `selectVariant`, records the
  pick in that beat's recent buffer, and returns the chosen line. Pure-ish and framework-free
  (no React); deterministic given seed + the order of `next` calls.
- Reset/recreate the director when a run begins, resumes, or restarts: seed it from `runSeed`
  in `startRun`, `hydrate`, and `goAgain`. Expose the current line / an `advance` action to the
  UI via the store or a thin context (no game state in context — director only).
- Implement `Teleprompter.tsx`: presents **one beat at a time**, large and paced (big type,
  the single line front-and-centre, an obvious advance control). It renders a line handed to
  it; it does not pick. Keep it dumb and reusable so each screen can drop it in.
- **Full-run no-repeat proof (headless):** in `director.test.ts`, drive a representative full
  run's beat sequence (briefing → N obstacles' clue/option/outcome beats + a scenario setup →
  push/run → getaway intro/countdown → win/bust sting) through the director and assert **no
  line repeats** within the run, and that two directors built from the **same seed** with the
  same call sequence emit an identical line sequence (determinism under seed).
**Acceptance:**
- Director is deterministic under seed (asserted) and surfaces no repeated line across a
  full-run beat walk (asserted).
- `Teleprompter.test.tsx` shows it renders the current beat's line and the advance control
  requests the next line; it never blocks (no disabled dead-end).
- Store test confirms the director re-seeds on `startRun`/`goAgain` and is rebuilt on
  `hydrate` from the saved seed.
- `npm run check:full` green.
**Depends:** E8.2

## Task E8.4 — Wire narration into the pre-Offer beats (briefing, obstacle, scenario)
**Files:** `src/console/screens/Briefing.tsx`, `src/console/screens/ObstacleRoom.tsx`, `src/console/screens/ScenarioRoom.tsx` and their co-located tests
**Do:**
- Replace the hardcoded `MANSION_LABELS` paragraph in `Briefing` with a director `briefing`
  line scoped by `mansion.type`, presented through `Teleprompter`. Keep the single "Begin"
  action and the `mansion-dressing` test hook.
- In `ObstacleRoom`, surface a director `obstacleClue` line scoped by the room's `gameId`/`lane`
  (the van's clue), and render each option's `optionDescription` line scoped by `greedy`
  (and lane if authored) alongside the existing game/reward/heat facts. The clue is the
  performed teleprompter beat; option descriptions are short labels on the cards.
- In `ScenarioRoom`, surface a director `scenarioSetup` framing line via the teleprompter
  before the scenario's own authored body text + the two choices.
- A beat is requested **once per room occurrence** (stable within a render so re-renders don't
  re-pick); the GM advancing the run drives the next pick. Narration is additive only — every
  existing action/override/commit control stays exactly as-is (no new blocking states).
**Acceptance:**
- Briefing, ObstacleRoom, and ScenarioRoom each render a director-selected line for their
  beat(s); existing flow/test hooks still pass.
- No screen gains a dead-end: all commit/choice/override controls remain reachable regardless
  of narration state (asserted by the existing + updated screen tests).
- `npm run check:full` green.
**Depends:** E8.3

## Task E8.5 — Wire narration into the outcome & endgame beats (resolve, offer, getaway, result)
**Files:** `src/console/screens/MinigameHost.tsx`, `src/console/screens/Offer.tsx`, `src/console/screens/Getaway.tsx`, `src/console/screens/Result.tsx` and their co-located tests
**Do:**
- On minigame resolution (`MinigameHost`, the `RESOLVE_MINIGAME` moment), surface a director
  `outcomeQuip` scoped by the chosen `outcome` (`clean | complication | botched`) — the
  comedic three-tier quip — via the teleprompter, without changing how the GM taps the outcome.
- In `Offer`, replace the hardcoded escape-signal hint / push-run copy with a director
  `pushRun` line scoped by a `heatBand` derived from current Heat vs the escape signal (so the
  "getting hot — we can roll" energy is selected, not fixed). Keep both push-on / call-getaway
  buttons exactly as they are.
- In `Getaway`, surface `getawayIntro` at the start and a `getawayCountdown` line for the
  finale tension, via the teleprompter, layered over the existing timer/counter UI (no change
  to scoring or timing).
- In `Result`, surface `winSting` or `bustSting` scoped by `session.present.win`, alongside the
  existing score breakdown and "go again".
- All of these are additive flavour; every existing action and GM override stays reachable.
**Acceptance:**
- Resolve, Offer, Getaway, and Result each render the correct director-selected beat scoped by
  outcome / heat band / win-state; existing test hooks and flow still pass.
- A run that reaches each of these beats surfaces lines distinct from earlier beats (the
  director's no-repeat guarantee, already proven in E8.3, holds end-to-end through the wired
  screens); no narration state blocks an action.
- `npm run check:full` green; `npm run content:validate` green.
**Depends:** E8.4
