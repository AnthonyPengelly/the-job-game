# Plan ŌĆö Epic E16: Crew identity, Setup & Result

This epic gives the run a **named crew**, replaces free-text quirks with a fixed table of **10 named
stat-combo quirks**, removes the bogus setup checklist, surfaces dice mode at Setup, fixes the
leaderboard to carry team name + actual banked cash, and rebuilds Setup and Result to match
`design-system/redesign/1 - Setup.html` and `9 - Result.html`.

The work is decomposed into five tasks layered bottom-up: (T1) the quirk table as content+schema+engine,
(T2) crew name threaded through engine state / save / non-Setup chrome, (T3) the leaderboard schema bump
+ data correctness, then the two UI rebuilds (T4) Setup and (T5) Result that consume all of the above.
T1 and T2 are independent and can build in parallel; T3 depends on T2; T4 depends on T1+T2+T3; T5
depends on T2+T3.

**Two spec notes for builders/reviewers (not blockers):**
1. **Quirk magnitudes follow the epic's numbers, not the mockup's sample data.** `docs/EPICS.md` E16 is
   explicit and self-consistent: each quirk is **+2 to one lane** (4 of them) **or +1 to two different
   lanes** (6 of them) = **10 quirks**. The roster shown in `1 - Setup.html` (Lucy/Millie/ŌĆ” with
   "Grease Monkey +1 TEC", "Cool Head ┬▒0") is illustrative placeholder data ŌĆö do not treat its numbers
   as the spec. Use heist-flavoured names; the dropdown shows each quirk's actual effect (e.g. "+2 TEC"
   or "+1 TEC / +1 STE"). The design reviewer judges layout/visual fidelity, not these numbers.
2. **Quirks are mandatory.** Design-doc line 55 / EPICS: every player starts with one. The dropdown
   always has a selection (default to the first quirk); drop the old "optional" notion.

## Task E16.1 ŌĆö Quirk table: content + schema + engine application
**Files:**
- `presets/default/content/quirks.json` (new)
- `src/content/schema/quirks.ts` (new), `src/content/schema/index.ts` (export)
- `src/platform/presets/browser.ts` (load + parse quirks.json into the bundle)
- `src/platform/presets/build-config.ts` (build a frozen quirk table into `EngineConfig`; add to `PresetBundle`)
- `src/engine/config.ts` (add `quirks` to `EngineConfig`)
- `src/engine/run.ts` (`makePlayer`/`startRun` apply the selected quirk's lane boosts to starting stats)
- `src/engine/reduce.ts` (thread `cfg` into the `startRun` call)
- Tests: `src/content/schema/quirks.test.ts` (new), `src/engine/run.test.ts` (boosts applied), and update any sim/build-config fixtures that construct an `EngineConfig`.

**Do:** Design the **10 named quirks** as preset data: 4 single-lane (+2 to one of tech/physical/charm/stealth)
and 6 two-lane (+1 to each of two distinct lanes ŌĆö all 6 lane pairs), each with a heist-flavoured name
and a stable `id` (kebab, branded `QuirkId` at the engine boundary). Add a Zod schema validating
structure (id, name, and a `boosts: { lane, magnitude }[]` or equivalent) and the "+2-single / +1+1-double"
shape. Load and freeze it into `EngineConfig.quirks` (a lookup keyed by quirk id). In the pure engine,
`startRun` must look up each player's `quirk` in `cfg.quirks` and **add** its boosts on top of
`MEDIOCRE_STATS` (unknown/absent quirk ŌåÆ mediocre baseline, never throw ŌĆö GM-override / no-dead-end).
Keep the engine pure (no content imports; reads only `cfg`). Determinism preserved (no RNG involved in
quirk application).

**Acceptance:** `presets/default/content/quirks.json` defines exactly 10 quirks matching the +2-single /
+1+1-double rule with distinct lane pairings; the schema rejects malformed quirk data (parse-at-boundary);
`startRun` applies the chosen quirk's boosts to a player's starting stats and leaves stats at baseline when
no/unknown quirk; engine stays framework-agnostic and deterministic; `npm run check:full` passes (unit +
sim + types + lint).
**Depends:** none (E14 gear/economy already merged).
