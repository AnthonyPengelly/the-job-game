# Plan — Epic E11: Tuning panel & preset tooling

**Goal (from `docs/EPICS.md`):** retune the game without leaving the app or editing code — a fully-realised preset system (named presets, clone, edit, select-at-boot, swap-without-rebuild), an in-app tuning panel with sliders for Heat / Getaway / scoring constants that embeds the Monte Carlo and shows run-length and win-rate distributions shifting live, and validation that rejects bad presets with a clear message.

**Current state of `main` (what already exists, do NOT rebuild):**
- **Preset schemas** — every pack has a Zod schema in `src/content/schema/` (`tuning.ts`, `scaling.ts`, `meta.ts`, content packs). `tuningSchema` already enforces ranges (e.g. `runAtFraction ∈ (0,1)`, positive exponents). Validation primitives exist; E11 wires them to an in-app flow.
- **Config build** — `src/platform/presets/build-config.ts` turns a `PresetBundle` into a frozen `EngineConfig`. `browser.ts` statically imports + parses the **bundled `default`** preset only; `load.ts` is the Node (fs) loader used by the sim harness.
- **Store** — `src/console/store/store.ts` (Zustand) holds a single fixed `cfg` injected at creation by `StoreProvider` via `loadDefaultConfig()`. Settings (`src/content/schema/settings.ts` + `src/platform/persistence/settings.ts`) currently carry only `diceMode`, versioned + Zod-parsed, under their own localStorage key. Leaderboard/save live under separate keys.
- **Monte Carlo** — `sim/balance.sim.ts` runs the headless Monte Carlo and asserts design targets A–J via `npm run sim:check`. Its simulation core (`simulateRun`/`computeCell`) and the crew policy (`sim/model-crew.ts`) depend **only on `@/engine` + an `EngineConfig`** (no Node, no `Math.random`) — i.e. already browser-safe; only the *preset loading* uses Node fs.

**Approach.** Build bottom-up. (1) A browser preset store: localStorage-backed CRUD for user presets (clone/edit/save/list/delete/select) that inherit non-tuning packs from the bundled `default`, plus a validate-then-build path that returns a typed ok/error result instead of throwing. (2) Wire the store + boot to swap the active preset at runtime. (3) Extract the existing Monte Carlo core into a shared, browser-safe module (single source of truth — the CI harness re-uses it) so the same simulation can run in-app. (4) Run it in a Web Worker at reduced N behind a distributions chart component. (5) Assemble the GM-only tuning panel that binds sliders to the editable preset, shows the live charts, and drives clone/save/select with clear validation messaging.

**Golden-rule notes for every task:** the engine stays pure and preset-driven (no new tunable constants in code); all new randomness in the sim draws from the seeded RNG already threaded through `simulateRun`; the tuning panel is **GM-only** and must never appear in / leak into `player-view`; every preset edit is GM-reversible (cancel/reset — no dead-ends); all loaded/persisted preset data is Zod-parsed at the boundary before use.

## Task E11.1 — Browser preset store: clone / edit / save / list / select + validate-then-build
**Files:**
- `src/platform/presets/preset-store.ts` (+ `preset-store.test.ts`) — localStorage CRUD for user presets.
- `src/content/schema/settings.ts` — add `activePresetId` (string, default `'default'`); bump `SETTINGS_VERSION`.
- `src/platform/persistence/settings.ts` — round-trip the new field (the existing reset-to-default path already covers the version bump; add a migration default so old saved settings still load).
- `src/platform/index.ts` — export the new preset-store API.
- Reuse existing `tuningSchema`, `buildConfig`, `loadDefaultConfig`/the parsed default bundle.

**Do:**
- Define a `UserPreset` shape (and Zod schema in `src/content/schema/`): `{ id, name, baseId: 'default', tuning: <full ParsedTuning> }`. User presets edit **tuning only**; scaling, content packs and banks are inherited from the bundled `default` bundle. (Document this scope in a comment — the panel edits Heat/Getaway/scoring, which all live in `tuning.json`.)
- localStorage-backed API under a **dedicated key** (e.g. `the-job:presets`, separate from save/leaderboard/settings so clearing presets never wipes a run): `listPresets()` (returns the read-only bundled `default` plus all user presets), `readPreset(id)`, `writePreset(p)`, `deletePreset(id)`, `clonePreset(fromId, newName)` (deep-copies the source tuning, mints a new id). All reads Zod-parse at the boundary; malformed stored presets are skipped/reset loudly (mirror `readSettings` behaviour), never crash the table.
- Add `buildConfigFromPreset(id)` returning a typed result `{ ok: true; cfg: EngineConfig } | { ok: false; error: string; path?: string }`: it `tuningSchema.safeParse`s the preset's tuning, composes it with the default bundle's other packs, calls `buildConfig`, and on any Zod failure returns a **clear human-readable message + offending path** (e.g. `tuning.heat.runAtFraction must be < 1`) instead of throwing.

**Acceptance:**
- Unit tests (in-memory `StorageLike`): clone produces an independent copy with a new id; write/read/list/delete round-trip; `listPresets()` always includes `default` and never lets a user preset overwrite it.
- `buildConfigFromPreset` on a valid edited tuning returns `{ok:true}` with a frozen `EngineConfig`; on an out-of-range value (e.g. `runAtFraction = 1.5`, negative ramp) returns `{ok:false}` with a message naming the field — **no throw**.
- Settings round-trips `activePresetId` and the version bump leaves existing dice-mode settings loadable (defaults applied). `npm run check:full` green.

**Depends:** none (E1/E10 already merged).

## Task E11.2 — Runtime preset swap in the store + select-at-boot
**Files:**
- `src/console/store/store.ts` — add `activePresetId` to state and an `applyPreset(id)` action.
- `src/console/store/StoreProvider.tsx` — resolve the active preset at boot.
- `src/console/store/store.test.ts` (+ a focused new test file if cleaner).

**Do:**
- Store: hold `activePresetId`. Add `applyPreset(id: string)` that calls `buildConfigFromPreset(id)`; on `ok` it swaps `cfg`, persists `activePresetId` to settings, and takes the **clean path** — clears the in-progress save and resets to a fresh pre-run state (Setup), since swapping rules mid-run is "allowed but flagged; a new run is the clean path" (`docs/CONTENT-AND-TUNING.md` §3). On `!ok`, leave state unchanged and expose the error for the caller (panel) to display.
- Boot selection order (`docs/CONTENT-AND-TUNING.md` §3): `?preset=` URL param → persisted `activePresetId` setting → `'default'`. `StoreProvider` resolves the id and builds the initial `cfg` from the preset store; if the resolved preset is missing/invalid, fall back to `default` and surface a one-time notice (reuse the existing `staleSaveNotice`-style pattern, or a new `presetFallbackNotice` flag).
- Keep `cfg` the single source of truth flowing through `reduce`; do not duplicate tuning anywhere in React state.

**Acceptance:**
- Tests: `applyPreset` to a valid user preset swaps `cfg` (assert a tuning value changed in the resulting config), persists the id to settings, and resets to a clean pre-run state; `applyPreset` to an invalid id leaves `cfg` untouched and reports an error.
- Boot honours `?preset=` and the persisted id; an unknown/invalid persisted id falls back to `default` with the notice set. Existing store/persistence tests stay green. `npm run check:full` green.

**Depends:** E11.1.

## Task E11.3 — Extract the shared, browser-safe Monte Carlo core (single source of truth)
**Files:**
- `src/console/tuning/montecarlo.ts` (+ `montecarlo.test.ts`) — the shared core: the crew policy (moved from `sim/model-crew.ts`) plus `simulateRun` and a `runMonteCarlo(cfg, opts)` extracted from `sim/balance.sim.ts`.
- `sim/balance.sim.ts` — refactor to import the core from `@/console/tuning/montecarlo` (it already resolves `@/` aliases); keep all A–J assertions and `BASE_SEED`/`N` exactly as-is.
- Delete/replace `sim/model-crew.ts` (its logic moves into the shared core; update any other importers).

**Do:**
- Move the crew policy and `simulateRun` (engine-only deps, no Node, no `Math.random`) into `src/console/tuning/montecarlo.ts`. The console may import engine runtime freely; placing the in-app Monte Carlo with the tuning panel keeps it in the aliased `src` tree. The CI harness lives outside `src` (unlinted) and already imports `@/platform`/`@/engine`, so importing `@/console/tuning/montecarlo` is fine and preserves **one body of code** (golden rule 2 / `docs/GAME-DESIGN-RIGOUR.md`).
- Expose `runMonteCarlo(cfg, { n, baseSeed, skill, headcount })` returning summary distributions usable by the panel: a **run-length histogram** (rooms/obstacles binned) and a **win-rate** number, per requested cell. Parameterise N and seed so the panel can run a fast reduced-N pass while CI keeps N=20_000.
- `sim/balance.sim.ts` computes its cells via the shared functions; assertions unchanged.

**Acceptance:**
- `npm run sim:check` still passes with the identical A–J assertions (this is the regression guard — the refactor must not move any number).
- New unit test: `runMonteCarlo` on the default `cfg` is deterministic (same seed ⇒ identical distributions) and a reduced-N call (e.g. N≈500) completes fast.
- `npm run check:full` green.

**Depends:** none (independent of E11.1/.2; do before E11.4).

## Task E11.4 — Monte Carlo Web Worker + live distribution charts
**Files:**
- `src/console/tuning/sim.worker.ts` — Web Worker wrapping `runMonteCarlo` (reduced N) off the main thread.
- `src/console/tuning/useMonteCarlo.ts` — hook that posts the current `EngineConfig`/tuning to the worker (debounced) and returns the latest distributions + a running/idle flag.
- `src/console/tuning/Distributions.tsx` (+ `Distributions.test.tsx`) — renders run-length and win-rate distributions from a result object (lightweight inline SVG/CSS bars; no CDN, no new chart dep unless already vendored — `docs/DESIGN-SYSTEM.md`).

**Do:**
- Worker runs the shared core at reduced N with a **fixed seed** so charts are deterministic and reproducible; main thread stays responsive while sliders move.
- `Distributions` takes a plain result object (so it is unit-testable without a worker) and shows the run-length histogram and per-skill/headcount win rates, styled with design-system tokens.
- Hook debounces recompute on tuning change and exposes a busy indicator.

**Acceptance:**
- `Distributions` renders histogram bars + win-rate readouts from an inline result object in a jsdom test (no worker needed in the test).
- A test on the core (from E11.3) shows distributions **shift** when tuning changes — e.g. hotter Heat (higher `escalation.ramp`/`obstacleHeat.greedy`) shortens run length and lowers win rate — proving the embedded Monte Carlo reflects edits.
- `npm run check:full` green; no CDN requests introduced (no-cdn sensor stays green).

**Depends:** E11.3.

## Task E11.5 — GM-only tuning panel: sliders, clone / save-as / select, live charts, validation messaging
**Files:**
- `src/console/tuning/TuningPanel.tsx` (+ `TuningPanel.test.tsx`) — the panel.
- `src/console/tuning/index.ts` — barrel export.
- `src/console/app.tsx` — mount the panel behind a GM-only toggle (a hidden settings screen, alongside `DiceModeControl`); **never** in `player-view`.

**Do:**
- Render sliders/number inputs for the tunable Heat constants, the Getaway curve, and scoring weights (the fields in `tuning.heat`/`escalation`/`obstacleHeat`/`outcomeHeat`/`getaway`/`scoring` per the `docs/CONTENT-AND-TUNING.md` §8 mapping table), bound to an **editable working copy** of the active preset's tuning. Editing updates the embedded `Distributions` (E11.4) live.
- Controls: **clone** the current preset to a new named user preset; **save** the working copy (via E11.1 `writePreset`); **select** a preset to play (drives E11.2 `applyPreset`, which persists + clean-resets). Each control validates through `buildConfigFromPreset`; an invalid edit shows a **clear inline message** (field + reason) and the preset cannot be selected/saved until fixed.
- No dead-ends (golden rule 1): a GM can always cancel/reset the working copy back to the loaded preset and close the panel. The bundled `default` is read-only (clone-to-edit).
- GM-only: assert by placement (console shell) and by not widening any `player-view` slice.

**Acceptance (epic gate, `docs/EPICS.md` E11):**
- Edit a preset's Heat constants in the panel → the distribution charts update live (test the binding: changing a slider triggers a recompute / new distributions).
- Save it as a new named preset (persisted to localStorage) and select it → `applyPreset` swaps `cfg` so a run plays on the new numbers — **all without a rebuild** (test the save→select→cfg-swap chain).
- An invalid edit is **rejected with a clear message** and cannot be selected (test surfaces the error text).
- Panel is GM-only — not reachable from / not leaking into `player-view`.
- `npm run check:full` green.

**Depends:** E11.2, E11.4.
