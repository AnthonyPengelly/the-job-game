# Plan — Epic E14: Reward economy: real-haul Loot, Gear every room, sell-don't-use

**Approach.** E14 makes rewards *feel* like a heist by changing three things that are all currently
absent on `main`: (1) Loot magnitudes are tiny single digits rendered with `.toLocaleString()`; (2)
obstacle options grant **Loot only** — gear comes solely from scenarios; (3) there is no way to decline
gear for cash. We build this bottom-up so each task is independently green: first a shared **display
formatter** (pure, no data change), then the **engine + schema mechanism** for gear-on-obstacle, then
the **preset-data rewrite** that both rescales Loot to real money *and* puts gear on most rooms, then
the **sell-don't-use** engine event + Spoils surface, and finally a **holistic balance pass** that lands
the design bands at the new magnitudes. Display formatting is a console concern; the engine keeps
storing integers and stays pure; every magnitude and the sell-value curve live in `presets/default` as
data (golden rule 3); the sell decision is an ordinary undoable engine event (golden rule 1).

The currency symbol is standardised on **`$`** (the redesign set uses `$`; the design-doc `£` flavour is
superseded for display, per the epic and design-doc v0.9 §1). Reward magnitudes target the figures the
mockups show: `$25k`, `$45k`, `$53k`, `$137k`, `$186k`.

Ordering rationale: E14.1 (formatter) is foundational and used by every later display change. E14.2 lands
the gear-on-obstacle *mechanism* (engine/schema/generation) so E14.3 can author the *data* against a
schema that accepts it. E14.4 (sell) needs gear to exist on rooms (E14.2) and the formatter (E14.1).
E14.5 (balance) runs last because it tunes the magnitudes set in E14.3 and accounts for gear-only options
and the sell flow.

## Task E14.1 — Central Loot formatter
**Files:** create `src/content/format.ts` + `src/content/format.test.ts`; edit
`src/console/shell/TopRail.tsx`, `src/console/screens/Spoils.tsx`, `src/console/screens/Result.tsx`,
`src/console/screens/Setup.tsx`, `src/console/overrides/OverridePanel.tsx` (and any other Loot readout
found by grepping `loot`/`Loot` in `src/console`).
**Do:** Add a single pure helper `formatLoot(n: number): string` producing exactly the shapes the
mockups use (`design-system/redesign/0 - Cockpit Shell.html`, `6 - Spoils.html`, `9 - Result.html`):
`$0`, `$5.6k`, `$53k`, `$137k`, `$1.2m` — integers under 1,000 as `$N`; thousands as `$X.Xk` (one
decimal, trailing `.0` trimmed, e.g. `$53k` not `$53.0k`); millions as `$X.Xm`. Replace **every** raw
Loot readout — `TopRail.tsx:100` (`loot.toLocaleString()`), `Spoils.tsx:168` (`{lootGained}`),
`Result.tsx` (`{state.loot}`), `Setup.tsx` leaderboard rows (`${(entry.loot/1000).toFixed(1)}k` and
`entry.score.toLocaleString()`), and the `OverridePanel` Loot section label — with `formatLoot`. Keep the
existing `sr-only` raw-integer span in `TopRail` (`data-testid="loot-total"`) so number-asserting tests
keep working; the visible `.v` uses `formatLoot`. The engine still stores integers — this is display only.
**Acceptance:** `formatLoot.test.ts` covers boundaries (`0`, `999`, `1000→$1k`, `5600→$5.6k`,
`53000→$53k`, `137000→$137k`, `1_200_000→$1.2m`, and a negative/guard case if any readout can show one);
every Loot readout in the console renders via `formatLoot` (no remaining `.toLocaleString()` on a loot
value); `npm run check:full` passes.
**Depends:** none.

## Task E14.2 — Gear-on-obstacle mechanism (engine + schema + generation)
**Files:** `src/engine/types.ts` (`ObstacleOption`), `src/engine/config.ts` (`ObstacleOptionConfig`),
`src/content/schema/room-templates.ts` (Zod), `src/engine/generation.ts` (carry gear onto the generated
option), `src/engine/reduce.ts` (`RESOLVE_MINIGAME` grants the option's gear), plus unit tests
(`src/engine/generation.test.ts`, `src/engine/reduce.test.ts`).
**Do:** Add an optional `gear?: GearGrantDescriptor` to `ObstacleOptionConfig`, its Zod schema, and the
runtime `ObstacleOption` type — reusing the **existing** `GearGrantDescriptor` model
(`powerUp` / `statBoost` / `bigScore`; single-lane `lane` or multi-lane `lanes`). This supports the three
reward shapes on `3 - Obstacle Room.html`: **Loot only** (`reward>0`, no `gear`), **Loot + Gear**
(`reward>0` + `gear`), **Gear only** (`reward:0` + `gear`). In `generateRoom`, copy `gear` onto the
generated `ObstacleOption` (spread like `commitRange`). In `RESOLVE_MINIGAME`, when the committed option
carries `gear` **and** the outcome is `clean` (the option's advertised reward fires on clean, mirroring
`reward`), push the gear onto `earnedGear` using the same single-lane→`GearId` / multi-lane→descriptor
resolution as `applyScenarioEffect` (extract/reuse the existing `resolveGearGrant` path in
`scenario.ts` rather than duplicating it). The gear then resolves through the existing Spoils/ToolRail
assign flow with no dead-end. Keep the schema's `[safe, greedy]` refinement intact.
**Acceptance:** unit tests prove: an option with single-lane `gear` grants the resolved `GearId` to
`earnedGear` on a clean outcome and nothing on complication/botched; a multi-lane `gear` carries the
descriptor unresolved; a **gear-only** option (`reward:0` + `gear`) grants gear and adds 0 Loot on clean;
the Zod schema accepts gear descriptors and still rejects a `[greedy, safe]` ordering; `npm run check:full`
and `npm run sim:check` pass (no preset data added yet, so the sim is unaffected).
**Depends:** none.

## Task E14.3 — Author real-money reward data (rescale Loot + gear on most rooms)
**Files:** `presets/default/content/roomTemplates.json`, `presets/default/content/scenarios.json`,
`presets/default/tuning.json` (`outcomeLoot`); update the exact-value tests that assert old single-digit
Loot (`src/console/screens/ObstacleRoom.test.tsx` `state.loot).toBe(1)/toBe(0)` and any sibling in
`reduce.test.ts`/`scenario.test.ts`/`generation.test.ts`).
**Do:** Re-scale **every** Loot source to real-money magnitudes on the order the redesigns show
(`$25k`–`$186k` across a run): each obstacle option `reward`, every scenario `lootDelta`
(`effect`/`roll.success`/`roll.failure`/`delayed.payoff`), and `tuning.outcomeLoot.complication` /
`outcomeLoot.botched`. Keep the safe<greedy reward ordering and relative ratios so the curve is unchanged
in shape — only the units grow (these are pure data; no magnitude is hardcoded in the engine). Then add
`gear` descriptors (using the E14.2 schema) to the **majority** of obstacle templates, mixing the three
shapes (Loot-only, Loot+Gear, Gear-only) so a full run reliably puts several gear cards on the table —
fixing "I never got offered gear." Spread gear lanes/kinds sensibly across templates (lane should suit
the template's `lane` tag where natural). Update the broken exact-value assertions to the new magnitudes
(prefer relative assertions like `toBe(option.reward)` over hardcoded literals where the test can read the
config).
**Acceptance:** `npm run content:validate` and the room-template/scenario schema tests pass; a clear
**majority** of obstacle templates carry a `gear` descriptor (assert via a content test counting
templates with gear); a played/simulated run banks dramatic figures rendered by `formatLoot`
(`$3.6k`/`$137k`, never `4`); updated unit tests pass; `npm run check:full` passes. (Landing the sim
*bands* at the new magnitudes is E14.5 — this task only needs the schema/content tests and check:full
green; if `sim:check` happens to drift here, E14.5 owns the re-tune.)
**Depends:** E14.2 (schema accepts gear), E14.1 (formatter for display verification).

## Task E14.4 — Sell-don't-use decision (engine event + preset curve + Spoils surface)
**Files:** `src/engine/types.ts` (new `SELL_GEAR` event in `RunEvent`), `src/engine/reduce.ts`
(handler), `src/engine/config.ts` + `src/content/schema/tuning.ts` + `presets/default/tuning.json`
(`gearSellValue` curve) + `src/platform/presets/build-config.ts` (wire the new tuning field into
`EngineConfig`), `src/console/screens/Spoils.tsx`, plus tests (`src/engine/reduce.test.ts`,
`src/console/screens/Spoils.test.tsx`).
**Do:** Add `{ t: 'SELL_GEAR'; index: number }` to `RunEvent`. Its reducer removes `earnedGear[index]`
(so a card can't be sold twice) and banks `loot += gearSellValue(state)` — a value from a **preset curve**
(`tuning.gearSellValue`, e.g. `{ base, perRoom }` scaling with run depth/`roomIndex` so later gear sells
for more), never a hardcoded constant. Add the curve to the tuning Zod schema and `EngineConfig`, default
it in `presets/default/tuning.json`, and expose a small pure helper (e.g. `gearSellValue(state, cfg)` in
`scenario.ts` or a new `gear.ts`) so both the reducer and the Spoils UI can show the figure. In
`Spoils.tsx`, each unassigned `GearCard` gains a **"Sell for $X"** button (X via `formatLoot`) beside the
assign affordance per `6 - Spoils.html`; clicking dispatches `SELL_GEAR` for that card's index. Because
it goes through the normal store `dispatch`, it is **undoable** like every mutation (golden rule 1) — no
special handling needed.
**Acceptance:** unit test — `SELL_GEAR` removes the targeted `earnedGear` entry and increases `loot` by
the preset curve's value for the current depth, leaving other state untouched; `UNDO_LAST` restores the
card and the Loot (assert via the store/undo path); the Spoils sell button renders the value via
`formatLoot` and dispatches the event; gear can still be **assigned** through the existing flow with no
dead-end; `npm run check:full` passes.
**Depends:** E14.1 (formatter), E14.2 (gear exists on rooms to sell).

## Task E14.5 — Balance re-tune & sim-harness alignment
**Files:** `presets/default/tuning.json` + `presets/default/content/roomTemplates.json` /
`scenarios.json` (fine-tune magnitudes only as needed to land bands), `sim/balance.sim.ts` (update any
numeric expectation that legitimately moves), `docs/GAME-DESIGN-RIGOUR.md` (refresh the documented design
targets if their numbers change), and `src/console/tuning/montecarlo.ts` only if the option-selection
policy needs to account for gear-only options.
**Do:** Run `npm run sim:check` against the re-scaled preset (now including Gear-only options that grant
0 Loot and the sell flow) and re-tune so **all** design bands hold at the new magnitudes: median ~4–5
obstacles (A), `P(rooms>10) ≤ 0.05` (B), run-length tightness ≥ 0.93 (C), win-rate bands by skill
`bad<avg<good` (D–G), score ratio `good ≥ 1.75× bad` / "Loot roughly doubles poor→good" (H), headcount
spread (I), and the structural checks (J). The win-rate/Getaway curve is unaffected by Loot units, and H
is a ratio, so most bands should hold under uniform scaling — but Gear-only options change the Loot the
sim banks, so verify and adjust. Update the harness's numeric expectations and the documented targets
**only** where a band's underlying number genuinely changed, with a comment justifying the new value
(matching the existing E1.7 sign-off comment style in `balance.sim.ts`).
**Acceptance:** `npm run sim:check` passes with assertions A–J green at the new magnitudes; `npm run
check:full` passes; the design targets in `docs/GAME-DESIGN-RIGOUR.md` match what the harness asserts; any
changed threshold carries a one-line justification comment.
**Depends:** E14.3, E14.4.
