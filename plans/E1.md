# Plan — Epic E1: Core rules engine

**State of `main` at planning time.** E0 is merged: the seeded RNG (`src/engine/rng.ts`),
the folder skeleton, path aliases, the import-direction lint sensor, and a green
`check:full` all exist. `presets/default/` already ships complete `_meta.json`,
`tuning.json`, and `scaling.json` (the exact numbers the engine must read — see
`docs/CONTENT-AND-TUNING.md` §8). **Nothing else of E1 exists**: `src/engine/index.ts`
is a placeholder, there is no `types.ts`/`reduce.ts`/`heat.ts`/`scoring.ts`/`generation.ts`,
no preset loader, no Zod schemas, and `npm run sim:check` is still the E0 stub. So this
plan builds the whole epic from the RNG up.

**Approach.** Port `docs/design/heat-model-simulation.py` to TypeScript as the *same code
that ships* (golden rule 2; `docs/GAME-DESIGN-RIGOUR.md` §2). The engine stays pure:
`reduce(state, event, config)` with all randomness flowing through the seeded RNG and all
tunable numbers injected from the active preset (never hardcoded — golden rule 3). The work
fans out as: foundation (types + RNG replay + a deterministic constructor) → a minimal
engine-facing preset loader (Zod-validated tuning/scaling/meta → an injected `EngineConfig`) →
the three pure rules modules (Heat, Getaway/scoring, generation) → the reducer that wires them
into the run state machine → the headless Monte Carlo (`sim:check`) that drives the *shipping*
`reduce` with a model-crew policy and asserts the design targets. The model-crew policy
(`SKILL` bands, `growth_bonus`, `player_bonus`) is a **harness input, not a preset field**
(`docs/CONTENT-AND-TUNING.md` §8 note) and lives only in the harness, never in `src/engine`.

**Scope boundaries (so builders don't sprawl into later epics).**
- **No GM-override events and no `UNDO_LAST`** — those are E2 (`overrides.ts`). E1's `RunEvent`
  union is exactly the in-run subset below, and E1's reducer `switch` is exhaustive over it with
  a `default: never`. E2 extends the union and the switch; the resulting compile errors are the
  intended guide rails, not a regression.
- **No gear/crew rewards, power-ups, exhaustion, or 2–7 scaling logic** — that is E2. E1 reads
  the `scaling.json` numbers it needs for the Getaway headcount term, but does not implement
  rotation/exhaustion/minCommit filtering.
- **Getaway resolves by odds, not by Articulate cards.** The reference model has no
  Heat→(target cards, timer) mapping; it computes Heat→odds directly and rolls. E1 implements
  exactly that (the balance-relevant part). The literal Heat→(target, time) table and the
  `cardsCleared/timeLeft` resolution are **E6** and need new preset fields that do not exist yet —
  E1 must not invent them. E1's `RESOLVE_GETAWAY` resolves win/bust from a seeded odds roll (with
  an optional explicit `win` seam for E6 to fill), mirroring the scenario-roll "externalRoll
  present → use it, absent → draw from RNG" pattern.
- **No real scenario content (the 44) and no minigame registry** — those are E7 and E4/E5. E1
  generation draws from a *minimal* `roomTemplates` pack it co-develops here (the table in
  `docs/CONTENT-AND-TUNING.md` marks `roomTemplates` as "E1/E3"); game ids are plain strings that
  match the keys already in `scaling.json#/minCommit`. The cross-pack "game id exists in registry"
  invariant cannot run until E4, and `content:validate` stays the E7/E10 stub.

**Determinism & replay design (applies to every task).** `RunState` carries an `rngState: number`
(the mulberry32 state word). `reduce` reconstructs an RNG from it, draws, and writes the advanced
state back — so `reduce` is a pure function of `(state, event, config)` and a run replays exactly
from its seed + event log. Persistence (`{ seed, eventLog }`) itself is **E3**; E1 only guarantees
the replay property and tests it.

## Task E1.1 — Domain types, RNG replay support, and the initial-state constructor
**Files:** `src/engine/types.ts` (new), `src/engine/rng.ts` (extend), `src/engine/run.ts` (new),
`src/engine/index.ts` (re-export), `src/engine/rng.test.ts` (extend), `src/engine/run.test.ts` (new)
**Do:**
- `types.ts`: the pure domain model — `Lane` (`tech|physical|charm|stealth`), `Outcome`
  (`clean|complication|botched`), `RunPhase` (`briefing|room|minigame|offer|getaway|result`),
  branded ids (`PlayerId`, `GearId`, `GameId`), `Player`, `CarriedEffect`, `RoomResult`,
  `MansionDressing`, `RunState` (incl. `seed`, `rngState`, `phase`, `heat`, `loot`, `crew`,
  `roomIndex`, `obstacleCount`, `currentRoom`, `carried`, `history`, `escapeSignal`), and the
  **E1 `RunEvent` union** keyed on `t`: `START_RUN`, `CHOOSE_OPTION`, `RESOLVE_MINIGAME`,
  `CHOOSE_SCENARIO`, `PUSH_ON`, `CALL_GETAWAY`, `RESOLVE_GETAWAY`. Also a harness-facing
  `Skill = 'bad'|'avg'|'good'` type (used by E1.7; lives in engine types as a shared vocab,
  but carries no tunable numbers). Match the names in `docs/CONVENTIONS.md` and the sketches in
  `docs/ARCHITECTURE.md` §4. No logic in this file.
- `rng.ts`: add `state(): number` to the `Rng` interface (returns the current internal `s`), and
  ensure `mulberry32(capturedState)` resumes the identical stream from that point. Add a
  `rngFromState(state: number): Rng` alias if it reads cleaner. Keep `Math.random` out (lint).
- `run.ts`: `initialState(seed: number): RunState` (minimal pre-briefing state; `rngState = seed >>> 0`)
  and `startRun(state, event): RunState` building the briefing state from a `START_RUN` payload
  (crew names/quirks as given; crew stats/power-ups default to the mediocre baseline — gear/boosts
  are E2). Deterministic: same seed + same setup ⇒ identical `RunState`.
**Acceptance:** typecheck/lint/`check:full` green; `rng.test.ts` proves `state()`+`mulberry32(state())`
reproduces the continuing stream exactly; `run.test.ts` proves `initialState`/`startRun` are
deterministic for a fixed seed and set `rngState`. No `any`/`@ts-ignore`/unjustified `as`.
**Depends:** E0.

  `runAtFraction`; `escalation`: `onsetRoom`, `rampPerObstacle`; `obstacleHeat`: `safe`, `greedy`,

  `getaway`: exponent/skillTerm/skillPivot/headcountTerm/clamp; `scoring`:

  `presets/<id>/_meta.json`, `tuning.json`, `scaling.json` from disk (Node `fs`, repo-root relative),
  Zod-parses each, composes a **frozen** `EngineConfig`. Content packs (scenarios/gear/etc.) are out
  of scope — they don't exist until later epics; the loader only needs tuning/scaling/meta for E1.
**Acceptance:** `loadPreset('default')` returns a frozen config whose numbers equal
`presets/default/*.json` (e.g. `hMax === 20`, `runAtFraction === 0.55`, `escalation.rampPerObstacle === 0.2`);
a deliberately malformed fixture throws a Zod error citing the offending path; typecheck/lint/test green.
This is the *minimal engine-facing loader only* — it does **not** wire `npm run content:validate`
(that stays the E7/E10 stub) and does not load content packs.
**Depends:** E1.1.

## Task E1.3 — Heat model (`heat.ts`)
**Files:** `src/engine/heat.ts` (new), `src/engine/heat.test.ts` (new), `src/engine/index.ts` (re-export)
**Do:** pure functions reading `EngineConfig`, porting the Heat arithmetic of
`heat-model-simulation.py` exactly:
- `obstacleDrip(roomIndex, cfg)` = `obstacleHeat.safe + floor(roomIndex * escalation.rampPerObstacle)`
  (the reference's `base_ob + int(room*ramp_step)` — this IS the escalation ramp; it grows with the
  room index and `escalation.onsetRoom` documents where it becomes material).
- `greedySurcharge(heat, cfg)` → adds `obstacleHeat.greedy` when a greedy option is taken; the
  "greedy is available while `heat < greedyBelowFraction * hMax`" predicate (`greedyAvailable`).
- `outcomeHeat(outcome, cfg)` → clean/complication/botched from `outcomeHeat.*` (0/1/2 in default).
- `applyScenarioSwing(heat, delta, cfg)` helper over `scenarioSwing.small`/`.big` (± values).
- `escapeSignal(state, cfg)` → `roomIndex >= 2 && heat >= runAtFraction * hMax` (Heat 11 by default).
- `forcedGetaway(heat, cfg)` → `heat >= hMax`.
- All results clamp `heat` at `>= 0` where the reference does (`H = max(0, H)`).
**Acceptance:** unit tests assert each function against the reference arithmetic — e.g. drip at
`roomIndex 5` = `1 + floor(5*0.2)` = `2`; greedy adds `+2` and only below `0.5*hMax`; botch outcome
heat `=== 2` (this is the structural side of `sim:check` assertion **J**); escape signal true at
Heat 11 / room ≥ 2, false at room 1; forced at Heat 20. Pure, no RNG, no tunable literals in code.
**Depends:** E1.1, E1.2.

## Task E1.4 — Getaway odds & scoring (`getaway.ts`, `scoring.ts`)
**Files:** `src/engine/getaway.ts` (new), `src/engine/scoring.ts` (new), `src/engine/getaway.test.ts`,
`src/engine/scoring.test.ts`, `src/engine/index.ts` (re-export)
**Do:** port `getaway()` and `resolve()`'s scoring from the reference:
- `getawayOdds(heat, cfg, headcount, crewSkill)`:
  `frac = heat/hMax; p = 1 - frac**getaway.exponent; p += (crewSkill - getaway.skillPivot)*getaway.skillTerm
  + scaling.profiles[headcount].getawayBonus * getaway.headcountTerm;` clamp to `getaway.clamp`
  (`[0.04, 0.97]`). `crewSkill` is a 0..1 number supplied by the caller (the harness passes the
  `SKILL` band; E2+ derives it from real crew stats — out of scope here).
- `resolveGetawayOutcome(state, cfg, { roll })` → win if `roll < getawayOdds(...)`; the caller
  (reducer) supplies a seeded `roll` so the engine stays pure.
- `scoreRun(loot, heat, win, cfg)`:
  win → `loot * (scoring.winBaseMultiplier + scoring.lowHeatStyleBonus * (1 - heat/hMax))`;
  bust → `loot * scoring.bustMultiplier`. (Reference: `loot*(1.0 + 0.5*(1 - H/HMAX))` and `loot*0.4`.)
**Acceptance:** `getawayOdds` reproduces the Python `getaway()` for a table of `(heat, headcount, skill)`
samples within a small float tolerance, incl. clamp endpoints; scoring matches the reference formulas
exactly (low-Heat style bonus is generous at Heat 0, zero at hMax); tests use the loaded default config,
no tunable literals in code.
**Depends:** E1.1, E1.2.

## Task E1.5 — Room/scenario stream generation + minimal `roomTemplates` pack & schema

`presets/default/content/roomTemplates.json` (new), `src/content/schema/room-templates.ts` (new),
`src/content/schema/index.ts` (extend), `src/platform/presets/load.ts` (extend to load roomTemplates),

**Do:**
- Author a **minimal** `roomTemplates.json`: an `obstacles` array (each template: a `gameId` string
  drawn from the ids already in `scaling.json#/minCommit`, a `lane`, and safe/greedy option shapes —
  each option carrying `{ greedy: boolean, heatCost, reward }`) and a `scenarios` array (each: a
  two-choice shape whose hidden effects are cool/loot/heat deltas expressed via `scenarioSwing`
  magnitudes). Enough entries that no-repeat draw is meaningful and testable. Add its Zod schema and
  extend `loadPreset` to parse it (still no `content:validate` CLI — E7/E10).
- `generation.ts` (pure, seeded via `rngState`): `generateRoom(state, cfg)` → produces the next room
  (obstacle option menu or scenario), drawing from `roomTemplates` **without repeats within the run**,
  and advancing `rngState`. Carried effects: tick `CarriedEffect`s each room (e.g. ticking-briefcase
  countdown, a 3-room unlock) and expire/fire them deterministically. Obstacle templates always expose
  both a safe and a greedy option so the run policy (and later the GM) has the push-your-luck choice.

  **E2/E5/E7**; E1 generation is the seeded structural stream only.
**Acceptance:** `generateRoom` is deterministic for a fixed seed; repeated draws never repeat a template
until the pool is exhausted (no-repeat test); carried effects tick and fire/expire on schedule;
`roomTemplates.json` parses against its schema; typecheck/lint/test green.
**Depends:** E1.1, E1.2.

## Task E1.6 — The reducer: `reduce(state, event, config)` run state machine
**Files:** `src/engine/reduce.ts` (new), `src/engine/reduce.test.ts` (new), `src/engine/index.ts`
(export `reduce`)
**Do:** the pure heart — an **exhaustive** `switch (event.t)` over the E1 `RunEvent` union with a
`default: never` assert (`docs/CONVENTIONS.md`). Branches:
- `START_RUN` → delegate to `startRun` (E1.1), then generate the first room (E1.5); phase → `room`
  (or `briefing` then `room`, builder's seam — keep replayable).
- `CHOOSE_OPTION` → record the committed option on `currentRoom`; phase → `minigame`.
- `RESOLVE_MINIGAME { outcome }` → apply obstacle drip + greedy surcharge (E1.3) + outcome heat;
  award loot per the reference (clean → 2 if the chosen option was greedy else 1; complication →
  +1 loot and +complication heat; botched → no loot, +botched heat); increment `obstacleCount`;
  clamp heat ≥ 0; phase → `offer`; set `escapeSignal` (E1.3).
- `CHOOSE_SCENARIO { choiceId, attemptedBy? }` → apply the template's hidden cool/loot/heat effect
  (E1.3/E1.5); phase → `offer`; set `escapeSignal`.
- `PUSH_ON` → advance `roomIndex`, tick carried effects, generate the next room; **if `forcedGetaway`
  (heat ≥ hMax)** route straight to `getaway` instead.
- `CALL_GETAWAY` → phase → `getaway`.
- `RESOLVE_GETAWAY { win? }` → resolve win/bust: if `win` supplied use it (the E6 seam for real
  cards/time), else roll the seeded RNG against `getawayOdds` (E1.4); compute score via `scoreRun`;
  record result; phase → `result`. Advance `rngState`.
- Purity: reconstruct RNG from `state.rngState`, draw, write back; never mutate the input `state`;
  no `Math.random`/`Date.now`/timers.
**Acceptance:** unit tests cover **every** reducer branch (golden rule 8) with fixed seeds and exact
assertions; a replayed event log reproduces the identical `RunState` (determinism); a botch never
terminates the run (it routes to `offer`, structural assertion **J**); the forced-Getaway path fires
at heat ≥ hMax; the exhaustiveness `never` holds (adding an unhandled `t` is a compile error).
`check:full` green.
**Depends:** E1.3, E1.4, E1.5.

## Task E1.7 — Balance harness `sim:check`: headless Monte Carlo + design-target assertions
**Files:** `sim/balance.sim.ts` (new — the Monte Carlo + assertions), `sim/model-crew.ts` (new — the
harness-only crew policy), `vitest.sim.config.ts` (new), `package.json` (replace the `sim:check`
stub; exclude `sim/**` from the default `test`/`check` glob), `sim/README.md` (new, brief)
**Do:** the TS port of `heat-model-simulation.py`'s Monte Carlo, driving the **shipping** `reduce`
(`docs/GAME-DESIGN-RIGOUR.md` §2, §4). Tooling note: there is no `tsx`/`ts-node` in the repo — run the
harness via a dedicated **vitest** config (`sim:check` = `vitest run --config vitest.sim.config.ts`),
which gives deterministic execution and non-zero exit on any failed assertion for free. Keep the sim
spec out of the normal `npm test` run.
- `model-crew.ts` (harness input, **not** preset, not `src/engine`): the `SKILL` bands
  (`bad 0.45 / avg 0.65 / good 0.82`), `growth_bonus(i)=min(0.12,0.015*i)`, `player_bonus(n)`, and a
  `nextModelEvent(state, rng, skill, config)` policy that reproduces the reference branching: ~0.6
  obstacle vs scenario; greedy while `heat < 0.5*hMax`; `outcome()` tiering from `p = min(0.95, base +
  growth_bonus(room))` with the `-0.1` greedy penalty; the scenario cool/loot branches; escape when
  `room ≥ 2 && heat ≥ run_at`; the `room > 40` safety cap. It emits the E1 events into `reduce`.
- `balance.sim.ts`: for each cell in `skill {bad,avg,good} × headcount {2,4,7}`, run `N` seeded runs
  (fixed `BASE_SEED`, `N` ~12k order of magnitude), aggregate, and **assert all targets in one pass**
  (accumulate failures, print observed-vs-target + seed + preset for each, then fail) per
  `docs/GAME-DESIGN-RIGOUR.md` §3 table A–J: median obstacles 4–5 (avg/n4); P(rooms>10) ≤ 0.05;
  run-length tightness; win bands bad≈.37 / avg≈.48 / good≈.57 within ±5pt; strict ordering
  `win_bad < win_avg < win_good`; `loot_good ≥ 1.8 × loot_bad`; headcount `0.03 ≤ win_7 − win_2 ≤ 0.12`;
  and the structural **J** checks (`outcomeHeat.botched === 2`, no terminal-on-botch path). Read the
  preset id from `PRESET` env (default `default`) via `loadPreset`.
**Acceptance:** `npm run sim:check` runs the headless Monte Carlo over the **default** preset and
passes all assertions (replacing the stub); it is deterministic (re-running gives identical results);
its numbers reproduce `heat-model-simulation.py` within the documented tolerances; `npm run check`
and `npm run check:full` still pass and do **not** run the (slow) sim spec.
*Risk note (not a blocker):* the default preset uses `runAtFraction = 0.55` (the canonical Heat-11
escape signal), whereas the reference `verify()` used `0.65`. If, after a faithful port, the default
preset misses a band, treat it as a **balance finding** per `docs/GAME-DESIGN-RIGOUR.md` §8 — surface
it (and, if it can't be reconciled within tolerance, `PIPELINE_BLOCKED`); do **not** silently retune
the sacred `default` preset or loosen the gate.
**Depends:** E1.6.
