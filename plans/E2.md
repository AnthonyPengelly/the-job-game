# Plan — Epic E2: Crew, gear, scaling & GM overrides

**Approach.** E1 shipped the pure run engine (Heat, generation, getaway, scoring) and a
preset loader that exposes only the numbers E1 needed. E2 builds the *crew* layer on top of
that same pure engine: the two reward layers (stacking +1/+2 stat boosts; the four
non-stacking lane power-ups), gear assignment, exhaustion/rotation, the 2–7 scaling profiles,
and the full GM-override API with `UNDO_LAST`. No application/UI code — everything here is
pure TypeScript in `src/engine`, validated data in `presets/` + `src/content/schema`, and the
preset loader in `src/platform/presets`. The Zustand store, HUD and drag-drop UI that consume
all of this are E3.

**The hard constraint — locked balance.** `npm run sim:check` is a human-signed-off Monte
Carlo (commit `41646aa`, "do not revert") that drives the shipping `reduce()` — including
`generateRoom` — at headcounts **2, 4 and 7** with assertions A–J. Any change that alters the
RNG draw order, the number of draws, or which templates/options are selected at those
headcounts will break locked assertions. The E2 reward/override/exhaustion events are never
dispatched by the sim's model crew, so they are balance-neutral by construction. The **one**
risk is generation (E2.4): it must be implemented as *pure annotation* that adds no RNG draw
and changes no template/option selection. Every design-bearing task keeps `sim:check` green.

**Architecture decisions used here (already in the docs, not invented):**
- *Undo is pure; the store holds the history* (ARCHITECTURE §5, §8). `reduce(state,event)` stays
  pure and history-free. A thin **pure session reducer** — `reduceSession(session, event, cfg)`
  over `SessionState = { present: RunState; past: RunState[] }` — implements `UNDO_LAST` by
  popping `past`. This is the engine-level, fully-testable construct the E3 Zustand store will
  wrap. `UNDO_LAST` is therefore a *session* event, not a member of the `reduce`-level
  `RunEvent` union (keeps `reduce`'s exhaustive switch clean). The design doc's
  "`RunEvent` … plus `UNDO_LAST`" is read as the session-level event set.
- *No dead-ends* (GOLDEN RULE 1): overrides go through the reducer like any other event and
  every tracked field is driveable to any legal value; the reducer never *rejects* a GM commit
  (e.g. `CHOOSE_OPTION` stays permissive even if a commit is the "wrong" size).
- *Everything tunable is preset data* (GOLDEN RULE 3): exhaustion strength, crew-per-option,
  dial curves and the gear catalog are loaded from the active preset, never hardcoded.

**Deferred (out of E2 scope, noted so builders don't gold-plate):**
- Feeding a crew-stats-derived `crewSkill` into the Getaway resolution is **E6's** job (the
  finale). E2 does not rewire `reduce`'s `RESOLVE_GETAWAY` (doing so would risk locked
  balance). `getaway.ts` already defaults `crewSkill` to `skillPivot`.
- The exact mechanical *magnitude/where-it-lands* of the 2–3-player "tired" penalty is "a
  tuning job for the build" (design §"Scaling 2–7"). E2 models exhaustion as **rotation**
  (rooms benched) driven by preset data, with `tired` → 0 rooms benched (a soft class, not a
  bench). The dial-side penalty hook is consumed when the dial is wired into mini-games (E4).
- Tracking individual held gear *items* per player is not in the design's `Player` model
  (it carries derived `stats` + `powerUps`). Gear assignment mutates those derived fields.

## Task E2.1 — Expose the full scaling profile in EngineConfig + loader
**Files:** `src/engine/config.ts`, `src/content/schema/scaling.ts`, `presets/default/scaling.json`,
`src/platform/presets/load.ts`, `src/engine/index.ts`, plus co-located tests
(`src/platform/presets/load.test.ts`, `src/content/schema/scaling.test.ts` — create if absent).
**Do:**
- Extend `EngineConfig.scaling` so the engine can read, per headcount string `"2".."7"`:
  `crewPerOption: [number, number]` and `exhaustion: 'full' | 'light' | 'tired'` (in addition
  to the existing `getawayBonus`). Also surface `dialCurve` (record keyed by gameId or
  `_default`, each `{ base, perLanePoint, tightenPerExtraCrew }`), `minCommit` (already present),
  `variant` (record of `{ soloVariantId?, variantId?, appliesAt }`), `excludedFromSolo: string[]`,
  and `soloEligibleMinPool: number`.
- Add a new **preset-data** field for exhaustion rotation strength: `exhaustionRest:
  Record<'full' | 'light' | 'tired', number>` (rooms benched per class). Add it to
  `scaling.json` with `{ full: 1, light: 1, tired: 0 }`, add it to `scalingSchema`
  (`src/content/schema/scaling.ts`, `.strict()`), and surface it on `EngineConfig.scaling`.
  Update the `_meta.units` note in `scaling.json` to describe it.
- Extend `loadPreset` to map the already-parsed scaling pack into the widened config (carry
  `crewPerOption`/`exhaustion` into `profiles`, plus the new top-level fields). Keep the result
  `deepFreeze`d.
**Acceptance:** `scalingSchema` validates the updated `presets/default/scaling.json`; a malformed
`exhaustionRest` is rejected (schema test); `loadPreset('default')` returns the new fields and they
are frozen; **`npm run check:full` and `npm run sim:check` both green** (config-shape change must
not alter any engine math — `getawayBonus` mapping unchanged). Pure data/plumbing only.
**Depends:** none (builds on E1).

## Task E2.2 — Gear catalog as data + the two reward layers (apply-gear) + `ASSIGN_GEAR`
**Files:** `src/content/schema/gear.ts`, `src/content/schema/index.ts`,
`presets/default/content/gear.json`, `src/engine/config.ts`, `src/platform/presets/load.ts`,
`src/engine/crew.ts` (new), `src/engine/types.ts`, `src/engine/reduce.ts`,
`src/engine/index.ts`, plus tests (`src/content/schema/gear.test.ts`, `src/engine/crew.test.ts`,
and a `RESOLVE`/`ASSIGN_GEAR` case in `src/engine/reduce.test.ts`).
**Do:**
- **Gear schema (data):** Zod `gearSchema` (`.strict()`, with a `_meta` block matching the
  established pack-meta pattern) describing a catalog of gear items, each
  `{ id, kind: 'statBoost' | 'powerUp', lane: Lane, magnitude?: number, flavour?: string[] }`.
  `statBoost` carries `magnitude` (1, or 2 for "Big Score"); `powerUp` carries no magnitude.
- **Gear data:** `presets/default/content/gear.json` with the mechanical reward set from
  `docs/design/heist-content.md` §GEAR: a `+1` stat boost per lane (tech/physical/charm/stealth),
  the rare `+2` "Big Score", and the four lane power-ups. Flavour-name lists may be included as
  data but are not required to be exhaustive (full banks are later content epics).
- **Config + loader:** add a `gear` catalog to `EngineConfig` (a `Record<string, GearDef>` keyed
  by gear id, or a typed list); `loadPreset` parses `content/gear.json` and surfaces it.
- **Engine reward logic:** `src/engine/crew.ts` with a pure `applyGear(player, def): Player`:
  `statBoost` → `stats[lane] += magnitude` (boosts **stack**); `powerUp` → `powerUps[lane] = true`
  (**no stack** — idempotent if already held). Returns a new `Player` (no mutation).
- **Reducer:** add `{ t: 'ASSIGN_GEAR'; gear: GearId; to: PlayerId }` to the `RunEvent` union and
  handle it in `reduce` — look the gear up in `cfg.gear`, apply it to the named player, return new
  state. Unknown gear id or player id throws (parse-at-boundary / loud failure), consistent with
  the existing reducer guards.
**Acceptance:** `gearSchema` validates the default gear pack; `loadPreset` exposes the catalog;
`applyGear` unit tests prove boosts stack (`+1` then `+1` ⇒ `+2`; `+2` Big Score), power-ups are
idempotent/no-stack, and the input player is not mutated; a `reduce` test shows `ASSIGN_GEAR`
applies the right effect to the right player and leaves others untouched. **`npm run check:full`
green; `npm run sim:check` green** (the sim never dispatches `ASSIGN_GEAR`).
**Depends:** E2.1 (loader pattern).

## Task E2.3 — Exhaustion / rotation
**Files:** `src/engine/crew.ts`, `src/engine/reduce.ts`, `src/engine/index.ts`, tests
(`src/engine/crew.test.ts`, cases in `src/engine/reduce.test.ts`).
**Do:**
- Pure helpers in `crew.ts`: `profileFor(headcount, cfg)` (resolves the scaling profile by
  `crew.length`, clamped to the `"2".."7"` keys), `isResting(player, roomIndex)` (true while
  `roomIndex < restingUntilRoom`), and `applyExhaustion(crew, committedIds, roomIndex, cfg)` —
  sets `restingUntilRoom = roomIndex + cfg.scaling.exhaustionRest[exhaustionClass]` on the
  committed players when that value is `> 0` (so `full`/`light` bench for the next room, `tired`
  does not bench). Returns a new crew array.
- Wire `applyExhaustion` into the `RESOLVE_MINIGAME` branch of `reduce`, using the obstacle's
  `committedBy`. This must add **no RNG draw** and must not touch `heat`/`loot`/`history`/`phase`
  (balance-neutral).
**Acceptance:** unit tests: for `n=5..7` (`full`) and `n=4` (`light`) the committed crew's
`restingUntilRoom` advances by the preset rooms count and `isResting` reads true next room; for
`n=2..3` (`tired`) no one is benched; benched/un-benched is reflected per `roomIndex`.
**`npm run check:full` green; `npm run sim:check` green** (RESOLVE_MINIGAME's measured
heat/loot/rooms/score outputs are unchanged).
**Depends:** E2.1.

## Task E2.4 — Scaling-aware "correctly-sized jobs" + dial-from-stat (pure annotation)
**Files:** `src/engine/scaling.ts` (new), `src/engine/generation.ts`, `src/engine/types.ts`,
`src/engine/index.ts`, tests (`src/engine/scaling.test.ts`, cases in
`src/engine/generation.test.ts`).
**Do:**
- Pure functions in `scaling.ts`:
  - `obstacleCommitRange(gameId, headcount, cfg): [number, number]` — `minCrew = max(minCommit[gameId],
    crewPerOption[headcount][0])`, `maxCrew = crewPerOption[headcount][1]` (clamped to headcount).
  - `resolveGameVariant(gameId, commitSize, headcount, cfg)` — returns the solo/2-player variant id
    when `cfg.scaling.variant[gameId].appliesAt` includes the commit size and the game is eligible,
    else the base id; respects `excludedFromSolo` + `soloEligibleMinPool` (a commit of 1 is only a
    legal "solo" slot when the obstacle pool size ≥ `soloEligibleMinPool` and the game is not in
    `excludedFromSolo`).
  - `computeDial(committedLaneRatings, gameId, headcount, cfg): number` from the game's
    `dialCurve` (or `_default`): `base + perLanePoint * Σ(lane ratings) + tightenPerExtraCrew *
    (commitSize − 1)` (or the curve's documented shape). Higher committed lane ratings lower the
    dial; more committed crew eases it.
- In `generateRoom`, **annotate** each generated `ObstacleOption` with its `commitRange`
  (`[minCrew, maxCrew]`) for the current `state.crew.length`. **Do not** change the scenario/obstacle
  decision, the template draw, the option construction order, or the number/sequence of RNG draws —
  this is additive metadata only (add the field to `ObstacleOption` in `types.ts`). The reducer's
  `CHOOSE_OPTION` stays **permissive** (no rejection of off-range commits — no dead-ends).
**Acceptance:** generation test — for `n = 2..7`, every dealt obstacle option's `commitRange` is
feasible (`1 ≤ minCrew ≤ maxCrew ≤ n`, and `minCrew ≥ minCommit[game]`); `excludedFromSolo` games
(`assemblyLine`, `defuse`) are never offered as a solo (commit-1) slot; `computeDial` strictly
decreases as a committed lane rating rises and eases with an extra committed crew member; results are
deterministic under a fixed seed. **Critically: `npm run sim:check` stays green** (no perturbation of
generation's RNG/selection at n=2/4/7) — and `npm run check:full` green.
**Depends:** E2.1.

## Task E2.5 — GM override events (set/adjust every tracked field; reroll/skip/jump)
**Files:** `src/engine/overrides.ts` (new), `src/engine/types.ts`, `src/engine/reduce.ts`,
`src/engine/index.ts`, tests (`src/engine/overrides.test.ts`).
**Do:**
- Add the GM-override members to the `RunEvent` union and apply them in `overrides.ts` (a pure
  `applyOverride(state, event, cfg): RunState`), delegated to from `reduce`'s switch:
  - `OVERRIDE_SET_HEAT { value }` / `OVERRIDE_ADJUST_HEAT { delta }` — clamp to `[0, hMax]`.
  - `OVERRIDE_SET_LOOT { value }` / `OVERRIDE_ADJUST_LOOT { delta }` — clamp `≥ 0`.
  - `OVERRIDE_SET_STAT { player, lane, value }` / `OVERRIDE_ADJUST_STAT { player, lane, delta }`
    (covers "hand or remove a stat boost").
  - `OVERRIDE_SET_POWERUP { player, lane, held: boolean }` ("hand or remove a power-up").
  - `OVERRIDE_SET_RESTING { player, untilRoom?: number }` (omit/clear ⇒ un-rest; set ⇒ rest).
  - `OVERRIDE_REROLL_ROOM` — regenerate the current room (advances `rngState` via `generateRoom`;
    this is a GM action, not a sim event, so it is balance-neutral).
  - `OVERRIDE_SKIP_ROOM` — advance `roomIndex` and generate the next room without resolving the
    current one (mirrors `PUSH_ON`'s generation, minus the resolution/forced-getaway short-circuit
    decision as appropriate).
  - `OVERRIDE_SET_PHASE { phase }` — jump to any `RunPhase`.
  - Gear grant/remove is expressed via the stat/power-up setters above (and the normal
    `ASSIGN_GEAR` reward path from E2.2); no separate gear-override event is required.
  - "Force a mini-game outcome" is already satisfied by the GM-supplied `RESOLVE_MINIGAME { outcome }`
    (the E3 stub picker); correcting an already-applied outcome is `UNDO_LAST` (E2.6) + re-resolve.
    Document this; do not add a redundant force-outcome event.
- Every override returns a fresh `RunState` (no mutation); all clamps respect the preset.
**Acceptance:** unit tests prove each override drives its field to any legal value from a reachable
state, with documented clamps (Heat `[0,hMax]`, Loot `≥0`); `reduce` remains exhaustive (the
`default: never` assert still compiles). **`npm run check:full` green; `npm run sim:check` green**
(overrides are never dispatched by the sim).
**Depends:** E2.2 (stats/power-ups), E2.3 (resting), E2.4 (room regeneration helpers if reused).

## Task E2.6 — Undo (pure session reducer) + `UNDO_LAST` + the no-dead-end property test
**Files:** `src/engine/history.ts` (new) (or extend `overrides.ts`), `src/engine/types.ts`,
`src/engine/index.ts`, tests (`src/engine/history.test.ts`, and the epic property test, e.g.
`src/engine/overrides.property.test.ts`).
**Do:**
- Define `SessionState = { present: RunState; past: RunState[] }` and a pure
  `reduceSession(session, event, cfg): SessionState` where `event: SessionEvent = RunEvent | { t:
  'UNDO_LAST' }`. Non-undo events: push `session.present` onto `past`, set `present =
  reduce(present, event, cfg)`. `UNDO_LAST`: pop `past` into `present` (no-op at the root — never a
  dead-end, never throws). Provide `initialSession(state)`. Optionally cap `past` depth via a preset
  or constant; if capped, document it.
- Export `reduceSession`, `initialSession`, and `SessionState`/`SessionEvent` from
  `src/engine/index.ts`. (The E3 Zustand store will hold a `SessionState` and call this; E2 ships
  only the pure function + tests.)
- **Property test (the epic acceptance gate):** generate many reachable `RunState`s by replaying
  random legal event sequences from seeds; then assert (a) for every tracked field — `heat`, `loot`,
  each player's `stats[lane]` and `powerUps[lane]`, `restingUntilRoom`, and `phase` — there exists an
  override event that drives it to an arbitrary chosen legal value (no dead-end); and (b) for every
  override applied via `reduceSession`, a following `UNDO_LAST` restores the **exact** prior
  `present` (deep-equal).
**Acceptance:** the property test passes (no-dead-end + undo-restores-prior-state, per the E2 gate);
`UNDO_LAST` at the root is a safe no-op; **`npm run check:full` green; `npm run sim:check` green**.
**Depends:** E2.5 (override events), E2.2/E2.3 (the fields exercised).
