# Plan — Epic E9: Audio engine & soundboard

**State of `main`:** none of E9 exists yet. There is no `src/platform/audio/`, no sound-manifest schema in `src/content/schema/`, no `presets/default/content/sound.json`, and no `src/console/soundboard/`. The shared `Metronome` primitive (`src/minigames/primitives/Metronome.tsx`) currently creates its own `AudioContext` and carries an explicit "E9 will integrate this with the shared audio engine" note. So the whole epic is genuinely absent and built from scratch, reusing the established patterns: content-as-Zod-data loaded at the boundary (mirroring narration), a leaf `platform` service the React surfaces drive (never the engine), and a React context that holds *only* theme/audio handles (architecture §5).

**Approach.** Build bottom-up along the dependency flow. First land the **sound manifest** as validated content + preset data + loader wiring (E9.1). Then the **Web Audio engine** in `platform/audio` — preload/play/mix, a master gain/mute, the Heat-nudged ambient bed, and a precise Web-Audio clock the metronome can schedule against (E9.2). Refactor the shared **Metronome** onto that precise clock and prove timing against the audio clock, not `setTimeout` (E9.3). Add the **audio-handle context + context-sensitive soundboard** mounted in the console shell (E9.4). Finally **wire the store's Heat to the ambient bed** so it drifts drone→heartbeat as Heat climbs, reacting to every change including GM overrides (E9.5). The three acceptance-gate items map 1:1 onto E9.3 (metronome accuracy), E9.4 (correct buttons per phase) and E9.5 (ambient responds to Heat), which keeps the epic QA pass straightforward.

A content note that is *not* a block: the repo ships no real audio assets. To stay green and honour "content is swappable data," E9.1 includes a small set of **placeholder** bundled audio files under `public/sound/` referenced by the manifest, and the engine (E9.2) tolerates any decode/load failure gracefully (logs once, never throws — no dead-end at the table). Real audio is a later content swap, not engine work.

## Task E9.1 — Sound manifest schema, default preset data & loader wiring
**Files:**
- `src/content/schema/sound.ts` (new — Zod schema + inferred types)
- `src/content/schema/index.ts` (export the schema + `ParsedSoundManifest` type)
- `src/content/schema/sound.test.ts` (new — schema validation tests)
- `presets/default/content/sound.json` (new — the default manifest)
- `presets/test-malformed-sound/content/sound.json` (new — a deliberately broken manifest for the loud-fail test, mirroring `presets/test-malformed-narration/`)
- `public/sound/` (new — a small set of placeholder bundled audio files referenced by the manifest, e.g. short/near-silent `.wav` per cue group)
- `src/platform/presets/load.ts` and `src/platform/presets/browser.ts` (add `loadSoundManifest` / `loadDefaultSoundManifest`, mirroring the narration loaders)
- `src/platform/index.ts` (export the new loader[s] + type)
- `src/platform/presets/load.test.ts` / `browser.test.ts` (cover the new loader path; assert the malformed manifest throws)

**Do:** Define a Zod schema for the sound manifest: a set of **cues**, each with a stable `id`, a `src` path (under `public/sound/`), a `channel`/group it belongs to (`ambient`, `heistSfx`, `sting`, `danger`, `finale` — the soundboard groups named in the epic), an optional `loop` flag (for ambient/danger beds), an optional default `gain`, and a `phases` list of `RunPhase` values declaring in which phase(s) the cue is *relevant* (drives the context-sensitive soundboard in E9.4). Separately model the **ambient bed**: the cues (or layered cue refs) used for the low-Heat drone and the high-Heat heartbeat so E9.5 can crossfade between them. Enforce unique cue ids (superRefine, like `narration.ts`). Author `presets/default/content/sound.json` with at least a few cues per group and the ambient drone/heartbeat pair, each pointing at a placeholder file actually present under `public/sound/`. Add `loadSoundManifest(id)` (Node/fs) and `loadDefaultSoundManifest()` (bundled import) mirroring `loadNarration`/`loadDefaultNarration`; the manifest is parsed at the boundary and returned separately — it never enters `EngineConfig` or the RNG stream.

**Acceptance:** A schema test passes: the default manifest parses, every `src` resolves to a file under `public/sound/`, every cue id is unique, every `phases` entry is a legal `RunPhase`, and the malformed preset fails loudly (throws) on load. `npm run check:full` green. No engine import of the manifest (import-direction sensor stays green).
**Depends:** none (builds on E7/E8 content + loader patterns already on `main`).

## Task E9.2 — Web Audio engine (preload, play/mix, master gain, ambient bed, precise clock)
**Files:**
- `src/platform/audio/engine.ts` (new — the `AudioEngine` factory/class)
- `src/platform/audio/clock.ts` (new — the precise Web-Audio scheduler/clock API, or co-located in `engine.ts` if small)
- `src/platform/audio/index.ts` (new — public surface)
- `src/platform/audio/engine.test.ts`, `clock.test.ts` (new — tests against a mock `AudioContext`)
- `src/platform/index.ts` (export the audio public surface)

**Do:** Implement a framework-free `platform/audio` service (no React; the console drives it). Given a `ParsedSoundManifest`, it: lazily creates/owns one `AudioContext` (with a `resume()` on first user gesture); **preloads** each cue's buffer via fetch + `decodeAudioData`, tolerating per-cue load/decode failure gracefully (log once, mark cue unavailable — never throw, no dead-end); exposes `play(cueId)`, `stop(cueId)` and per-**channel** + **master** gain control plus a master `mute`/`setMasterGain`. Implement the **ambient bed**: `setAmbient(intensity: 0..1)` crossfades the looping drone (low) toward the heartbeat (high) using `GainNode` ramps — the engine only consumes a normalized intensity; the Heat→intensity mapping lives console-side (E9.5) so no tunables are hardcoded here. Expose a **precise scheduler/clock** built on `audioContext.currentTime` with a lookahead window (the lookahead-loop pattern already sketched in `Metronome.tsx`) that E9.3 will consume — `now()`, `scheduleAt(time, cb)`/beat scheduling, `start()/stop()`. Provide a clean no-AudioContext fallback so headless/test environments and unsupported browsers don't crash.

**Acceptance:** Unit tests with a mock/stub `AudioContext` cover: manifest preload (incl. a cue whose decode rejects → engine still usable, cue marked unavailable); `play`/channel/master gain routing; `mute` silences master; `setAmbient(0)`→drone, `setAmbient(1)`→heartbeat, mid-values crossfade (assert relative gain values); scheduler fires scheduled callbacks in order against the (mock) audio clock. `npm run check:full` green; engine layer untouched (import-direction sensor green).
**Depends:** E9.1

## Task E9.3 — Shared Metronome on the precise audio clock (+ accuracy test)
**Files:**
- `src/minigames/primitives/Metronome.tsx` (refactor to schedule off the E9.2 audio clock instead of its own `AudioContext`)
- `src/minigames/primitives/Metronome.test.tsx` (extend — clock-accuracy test against the audio clock, not `setTimeout`)
- `src/minigames/primitives/index.ts` (only if the handle/options surface changes)
- Beat 16 / Follow the Circuit consumers (only the minimal wiring needed to pass the audio handle through; do **not** redesign those games)

**Do:** Rework `useMetronome` so beat scheduling rides the precise Web-Audio clock provided by the `platform/audio` engine (passed in via the audio handle / an injected engine), removing the primitive's private `AudioContext` ownership. Keep the existing handle shape (`onBeat`, `mute`, the `isBeatAudible` pure predicate) and the graceful fallback when no audio engine/context is available. Preserve determinism of *beat numbering* (beats are integer-indexed; audio timing is wall-clock by nature but beat callbacks must stay ordered and gapless). Make the two metronome games (Beat 16, Follow the Circuit) consume the shared engine handle rather than spinning up audio themselves.

**Acceptance:** A timing test asserts beats are scheduled at the correct inter-beat interval **measured against the (mock) audio clock's `currentTime`**, explicitly not via `setTimeout`/`vi.useFakeTimers` on wall time — this is the epic gate's "metronome timing accurate (test against audio clock, not setTimeout)". `isBeatAudible` semantics and the audible-beat-limit behaviour remain covered. Beat 16 and Follow the Circuit still run inside the loop. `npm run check:full` green.
**Depends:** E9.2

## Task E9.4 — Audio-handle context + context-sensitive soundboard
**Files:**
- `src/console/audio/AudioProvider.tsx` (new — React context that owns the `AudioEngine`, loads the default manifest, preloads, resumes the context on first gesture; holds *only* the audio handle per architecture §5)
- `src/console/audio/index.ts`, `useAudio` hook
- `src/console/soundboard/relevantCues.ts` (new — **pure** `(phase, manifest) => cueGroups` deriver)
- `src/console/soundboard/Soundboard.tsx` (new — renders only the buttons relevant to the current phase, grouped ambient/heist SFX/stings/danger/finale; clicking drives the engine; includes a master mute/volume control)
- `src/console/soundboard/index.ts`
- `src/console/soundboard/relevantCues.test.ts`, `Soundboard.test.tsx` (new)
- `src/console/app.tsx` (wrap the shell in `AudioProvider`; mount `<Soundboard>` alongside the HUD/override surface)

**Do:** Add an `AudioProvider` that instantiates and owns the audio engine, loads + preloads the bundled default manifest, and exposes it through context (React Context is for theme/audio handles only — never game state). Write a **pure** deriver that, given the current `RunPhase` and the manifest, returns the cue groups whose `phases` include that phase — this is unit-testable without React. The `Soundboard` reads the current phase from the store, renders only the relevant grouped buttons (ambient / heist SFX / stings / danger / finale), and on click calls `engine.play(cueId)` (looping beds toggle). Include a master mute / volume control so the GM is never stuck with sound on (no dead-end). Mount it in the console shell next to the HUD and override panel; it must surface only for an in-run state (mirror the `crew.length > 0` guard).

**Acceptance:** Gate item "soundboard surfaces correct buttons per phase": the pure deriver test proves each `RunPhase` yields exactly the cues whose manifest `phases` include it (e.g. finale cues only in `getaway`, danger cues only at the relevant phases); the `Soundboard` test renders for a couple of phases and asserts the visible button set matches and that clicking invokes the engine. Master mute toggles engine mute. `npm run check:full` green; console does not import `player-view`; this surface is GM-only by definition so no GM-only state leaks.
**Depends:** E9.1, E9.2

## Task E9.5 — Wire Heat to the ambient bed
**Files:**
- `src/console/audio/useAmbientBed.ts` (new — subscribes to store `heat` + `cfg.heat.hMax`, maps to a 0..1 intensity, calls `engine.setAmbient`)
- `src/console/audio/AudioProvider.tsx` or `src/console/app.tsx` (mount the hook at app root so it tracks the live run)
- `src/console/audio/useAmbientBed.test.ts` (new)

**Do:** Add a small hook, mounted once at the console root, that reads `heat` from the store and `heat.hMax` from the active `EngineConfig`, computes a normalized intensity (e.g. `clamp(heat / hMax, 0, 1)`, low→drone, high→heartbeat), and calls `engine.setAmbient(intensity)` whenever Heat changes — **including** changes from GM overrides and `UNDO_LAST`, since those flow through the same store `heat`. No Heat→intensity constant is hardcoded in the engine; the mapping lives here and reads `hMax` from the preset. Ensure the bed starts/stops with the run lifecycle and respects master mute.

**Acceptance:** Gate item "ambient bed responds to Heat": a test drives the store from low to high Heat (and via an override) and asserts `engine.setAmbient` is called with a rising intensity (~0 at Heat 0, ~1 near `hMax`), and that `UNDO_LAST`/override-down lowers it again — proving no dead-end. `npm run check:full` green.
**Depends:** E9.2, E9.4
