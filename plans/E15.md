# Plan — Epic E15: Heat-scaled difficulty & loot progression

**Approach.** E15 makes push-your-luck *bite*: as Heat climbs and the run deepens, mini-games get
harder and the Loot/Gear on offer gets richer, and a crew that hasn't levelled the relevant lanes can
no longer stay clean late. Three facts on `main` shape the work: (1) obstacle difficulty
(`computeDial`) and scenario DC (`computeDC`) depend **only** on committed lane ratings — there is no
Heat/depth term; (2) rewards are **flat** (`option.reward` is copied verbatim at generation, granted
as-is in `RESOLVE_MINIGAME`); (3) **the balance sim is decoupled from `computeDial`** — `montecarlo.ts`
derives obstacle outcomes from an abstract skill probability `p = skill + playerBonus(n) +
growthBonus(room)`, and `growthBonus` makes *later* rooms *easier* (the opposite of E15's intent),
while scenario rolls *do* run through the real `computeDC`. The sim also has no per-room clean-rate
instrumentation and no "levelled vs un-levelled" dimension, so today it cannot express the shape E15
must assert.

We build bottom-up so every task is independently green. First add the **Heat/depth difficulty
mechanism** (a term in the obstacle dial + scenario DC, driven by preset curves that **default to a
no-op** so the sim stays green); then the **Heat/depth reward-scaling mechanism** (a preset curve
applied at generation, also default no-op); then **rework the sim model + instrumentation** (heat/depth
penalty in the obstacle `p`, a levelling dimension, per-room-band clean-rate in the result) while
preserving the existing A–J bands at the no-op defaults; finally a **balance pass** that sets the real
curve values, adds the new "early forgiving / late level-dependent" assertions, and re-tunes so every
band holds. All tunables are preset data (golden rule 3); the engine stays pure (golden rule 2); every
new value is GM-editable through the existing override surface and is undoable (golden rule 1); nothing
hardcodes a magnitude in the engine.

Ordering rationale: E15.1 (difficulty mechanism) and E15.2 (reward mechanism) are independent
preset-curve additions, each shipping as a behavioural no-op so `sim:check` stays green. E15.3 reworks
the simulation model and reads the E15.1 difficulty curve, so it depends on E15.1; it keeps existing
bands green by preserving aggregate behaviour at the no-op defaults and adds (but does not yet assert)
the new instrumentation. E15.4 turns the curves on, adds the new assertions, and re-tunes — so it
depends on all three.

## Task E15.1 — Heat/depth difficulty term in the obstacle dial and scenario DC
**Files:** `src/engine/scaling.ts` (`computeDial`), `src/engine/scenario.ts` (`computeDC`),
`src/engine/config.ts` (`scaling.heatDial`, `scenario.heatDC` fields), `src/content/schema/scaling.ts`
and `src/content/schema/tuning.ts` (Zod), `src/platform/presets/build-config.ts` (wire the new fields),
`presets/default/scaling.json` + `presets/default/tuning.json` (default curves = **no-op**),
`src/console/screens/MinigameHost.tsx` (pass Heat + roomIndex into `computeDial`),
`src/engine/reduce.ts` (`CHOOSE_SCENARIO` passes Heat + roomIndex into `computeDC`); plus unit tests
(`src/engine/scaling.test.ts`, `src/engine/scenario.test.ts`).
**Do:** Add a preset-driven Heat-and-depth contribution to **both** difficulty functions, *on top of*
the existing stat dial / lane-rating term — cool/early ⇒ easier, hot/deep ⇒ harder.
- Add a global curve `scaling.heatDial: { perHeat: number; perRoom: number }` and extend `computeDial`
  to accept the current `heat` and `roomIndex` (e.g. a `ctx: { heat, roomIndex }` arg) and add
  `heatDial.perHeat * heat + heatDial.perRoom * roomIndex` to the returned dial level (higher dial =
  harder). `MinigameHost.tsx` already has `present.heat` / `present.roomIndex` — thread them in. The
  GM-only `DialReadout` renders `dial.level`, so it becomes honest automatically; add a comment/note
  confirming the combined value is what's displayed.
- Add `scenario.heatDC: { perHeat: number; perRoom: number }` and extend `computeDC` to add
  `round(heatDC.perHeat * heat + heatDC.perRoom * roomIndex)` to the raw DC **before** the `dcClamp`
  (so a hot/deep roll needs a higher d20). `reduce.ts` `CHOOSE_SCENARIO` (~line 165) has `state.heat`
  and `state.roomIndex` — pass them; the stored `PendingRoll.dc` then reflects the combined value so
  the odds/`successOdds(dc)` shown to the GM stay honest. (The dramatic roll-reveal redraw is E19; here
  we only keep the number correct.)
- **Default both curves to `0` / `0` in the preset** (`heatDial`, `heatDC`) so this task is a pure
  behavioural no-op — `sim:check` is unaffected. Real values land in E15.4.
**Acceptance:** unit tests prove that with a non-zero `heatDial`, `computeDial` rises monotonically with
both `heat` and `roomIndex` and still adds the existing lane/commit terms; with a non-zero `heatDC`,
`computeDC` rises with `heat`/`roomIndex` and remains clamped to `dcClamp`; at the shipped default
curves (`0`) both functions return exactly today's values (regression test). Schemas accept and
validate the new fields; `npm run content:validate`, `npm run sim:check` and `npm run check:full` all
pass (no behaviour change at defaults).
**Depends:** none.

## Task E15.2 — Heat/depth reward-scaling mechanism at generation
**Files:** `src/engine/config.ts` (`rewardScale` field), `src/content/schema/tuning.ts` (Zod),
`src/platform/presets/build-config.ts` (wire it), `presets/default/tuning.json` (default curve =
**no-op**), `src/engine/generation.ts` (apply the curve to each generated option's `reward`), plus unit
tests (`src/engine/generation.test.ts`).
**Do:** Add a preset curve `rewardScale: { perHeat: number; perRoom: number }` (or an equivalent
`base + perHeat*heat + perRoom*roomIndex` multiplier shape) and apply it in `generateRoom` so the
**Loot on offer per obstacle option rises with Heat/room index** — the late-run greedy door becomes
genuinely tempting. Compute a multiplier `m = 1 + perHeat*state.heat + perRoom*state.roomIndex` and set
each generated `ObstacleOption.reward = round(template reward × m)` (the engine stores integers; this is
the offered magnitude, visible on the option card and granted on a clean outcome by the existing
`RESOLVE_MINIGAME` path). Gear richness rides the same wave through the E14 reward fields — if an option
carries a `bigScore` gear descriptor, scale its value by the same `m`; leave `powerUp`/`statBoost`
descriptors unscaled (their "richness" is the gear itself, which already concentrates in later rooms via
content). Keep the safe<greedy reward ordering intact (uniform multiplier preserves it). **Default the
curve to a no-op** (`perHeat: 0, perRoom: 0` ⇒ `m = 1`) so the task ships green; real values land in
E15.4. Generation runs at `PUSH_ON` with the already-incremented `roomIndex` and current `heat`, so the
scaling reads the room the crew is about to enter.
**Acceptance:** unit tests prove that with a non-zero `rewardScale` the generated option rewards
increase with `heat`/`roomIndex`, that safe<greedy ordering is preserved, and that `bigScore` gear
values scale while `powerUp`/`statBoost` descriptors do not; at the shipped default curve the generated
rewards equal the template values exactly (regression). Schema validates the field; `npm run
content:validate`, `npm run sim:check` and `npm run check:full` pass (no behaviour change at defaults).
**Depends:** none. (Reuses the E14 `gear` descriptor on `ObstacleOption`, already merged.)

## Task E15.3 — Sim model rework: Heat/depth penalty, levelling dimension, per-room clean-rate
**Files:** `src/console/tuning/montecarlo.ts` (model + `MonteCarloResult` + `MonteCarloOpts`), plus its
tests (`src/console/tuning/montecarlo.test.ts` if present, else add one). Do **not** add new
`sim:check` assertions here (that is E15.4).
**Do:** Make the simulation able to express Heat/depth-scaled difficulty and levelling, while keeping
the existing A–J bands green at the no-op default curves.
- **Heat/depth penalty in the obstacle `p`.** Today `rollOutcome(rng, p)` decides the obstacle outcome
  from a skill-only `p` that *ignores* `computeDial`. Add a Heat/depth difficulty penalty to the
  obstacle `p`, derived from the **same preset curve** the engine dial uses
  (`cfg.scaling.heatDial`, read off `state.heat`/`state.roomIndex`) mapped to a probability via a single
  documented model factor (a harness constant, like the existing `SKILL_VALUES`/`growthBonus` — the
  *tunable* stays in the preset, the harness owns the dial-level→probability interpretation). At the
  default `heatDial = 0` the penalty is 0, so aggregate behaviour is unchanged.
- **Levelling dimension.** Rework the free `growthBonus(room)` (which currently eases *every* late room
  regardless of investment) into a **levelling term** gated by a new `levelled` profile on
  `MonteCarloOpts`/`simulateRun` (e.g. `levelled: boolean` or a levelling rate): a *levelled* crew
  earns the room-growth offset (proxy for spending gear on the relevant lanes), an *un-levelled* crew
  gets little/none. Choose the default so the existing A–J cells (which pass `skill`/`headcount`) keep
  their current effective `p` and bands — i.e. the existing cells run at the "levelled" profile, so D–J
  are unchanged. Document the mapping in comments matching the existing port-from-Python style.
- **Per-room-band clean-rate instrumentation.** Extend `RunStats`/`MonteCarloResult` to record obstacle
  **clean-rate by room band** (e.g. early = first ~2 obstacles, late = obstacle ≥ a configurable index),
  accumulated across runs, so E15.4 can assert the shape. Add the fields without breaking existing
  consumers (the tuning panel + `balance.sim.ts`).
**Acceptance:** with the shipped default curves (`heatDial = 0`), `runMonteCarlo` reproduces today's
A–J numbers within the existing tolerances (assert via the unchanged `sim/balance.sim.ts` — it still
passes) and the new clean-rate-by-band fields are populated; a focused unit test shows that with a
non-zero `heatDial` the late-band clean-rate falls for an *un-levelled* crew and holds for a *levelled*
crew, and that the early-band clean-rate exceeds the late-band when the penalty is on; `npm run
sim:check` and `npm run check:full` pass.
**Depends:** E15.1 (reads `cfg.scaling.heatDial`).

## Task E15.4 — Balance: turn on the curves, assert the new shape, re-tune all bands
**Files:** `presets/default/scaling.json` + `presets/default/tuning.json` (set real `heatDial`,
`heatDC`, `rewardScale` values, fine-tune as needed), `sim/balance.sim.ts` (add new assertions K/L,
update any A–J threshold that legitimately moves with a justification comment),
`docs/GAME-DESIGN-RIGOUR.md` (refresh the documented target table to match).
**Do:** Set the difficulty and reward curves to real (non-zero) values and re-tune the default preset so
the full design intent holds and is asserted:
- **Difficulty rises with Heat/depth:** tune `heatDial` / `heatDC` so the `DialReadout` visibly hardens
  as Heat rises (verify the combined dial value grows) and late-room games are meaningfully harder than
  early ones.
- **Reward rises with Heat/depth:** tune `rewardScale` so the late greedy door offers materially more
  Loot than an early one (the redesign's "harder games for bigger numbers").
- **New `sim:check` assertions:** **K — early-room clean-rate high** (early-band clean-rate ≥ a target
  threshold); **L — late-room clean-rate is Heat/level-dependent** (late-band clean-rate for an
  *un-levelled* crew falls clearly below a *levelled* crew's late-band clean-rate, and below its own
  early-band rate). Add the cells these need (e.g. a levelled-vs-un-levelled pair at avg/n4).
- **Re-assert the existing bands at the new magnitudes:** A (median 4–5 obstacles), B (P(rooms>10) ≤
  0.05), C (run-length tightness ≥ 0.93), D–G (win-rate-by-skill ordering bad<avg<good within bands),
  H (score ratio good ≥ 1.75× bad), I (headcount spread), J (structural). Adjust curves until all hold;
  update any threshold whose underlying number genuinely shifts, each with a one-line justification
  comment in the E1.7/E14.5 sign-off style.
- Update `docs/GAME-DESIGN-RIGOUR.md`'s target table so the documented targets match what the harness
  asserts (including the new K/L rows).
**Acceptance:** with the default preset, the dial demonstrably hardens with Heat and the offered Loot
grows with it; `npm run sim:check` passes with assertions A–L green — early rooms forgiving, late rooms
punishing, an un-levelled crew degrading late while a levelled crew sustains, and **all existing design
bands still holding**; `docs/GAME-DESIGN-RIGOUR.md` matches the harness; any changed threshold carries a
justification comment; `npm run check:full` passes.
**Depends:** E15.1, E15.2, E15.3.
