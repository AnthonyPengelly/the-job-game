# Plan — Epic E6: The Getaway finale

**Approach.** Most of the Getaway's *resolution* already exists from E1–E3: the engine has `getawayOdds`, `resolveGetawayOutcome`, `scoreRun`, the `RESOLVE_GETAWAY { win? }` event (with both a seeded-roll path for the sim and a GM-supplied `win` seam for the real table game), and the console has a three-button `GetawayStub` outcome picker. What is genuinely missing is the **live finale itself**: the Heat→(target cards, timer) difficulty mapping that makes "low Heat generous, high Heat brutal" real, a referee screen that runs the round-the-circle Articulate game (counter, countdown, ditch-for-Heat, gear-spend affordances) and feeds the win/bust back to the engine, and the optional player-facing display on the isolated player-view surface.

The work splits into three dependency-ordered tasks: (1) the pure engine mapping + the `GETAWAY_DITCH` event + their preset tuning; (2) the live console Getaway screen that consumes the mapping and replaces the stub; (3) the player-view Getaway display fed by a new channel slice. The engine stays pure (no timers/audio — the countdown is ephemeral UI state in the console, never in `RunState`); all new numbers are preset data; the win/bust still flows through the existing `RESOLVE_GETAWAY { win }` seam, so scoring and the style bonus are untouched. Narration and sound are left as light, clearly-marked seams — full integration is E8/E9's job per the epic's "coordinate with E8/E9", and the E6 acceptance gate does not require them.

**Design reading (noted, not a block).** The content library (`docs/design/heist-content.md` §GEAR, mirrored in `presets/default/content/gear.json`) defines only `statBoost` and `powerUp` gear — there is **no consumable "spend-in-Getaway" gear type**. So "gear-spend (skip a card, buy seconds)" is implemented as **GM referee affordances** on the screen (the GM enforces the physical gear cost at the table; the app adjusts its timer/counter), consistent with golden rule #1 (GM in charge, app assists) and the design's "the app times, targets, scores; does not store cards." This does not contradict the docs and the acceptance gate does not require gear consumption to be modelled in engine state.

## Task E6.1 — Engine: Heat→(target, timer) Getaway brief + ditch event + tuning
**Files:**
- `src/engine/getaway.ts` (add `getawayBrief`)
- `src/engine/getaway.test.ts` (new or extend)
- `src/engine/config.ts` (extend the `getaway` config shape)
- `src/engine/types.ts` (add `GETAWAY_DITCH` to `RunEvent`)
- `src/engine/reduce.ts` (handle `GETAWAY_DITCH`)
- `src/engine/index.ts` (export `getawayBrief` if not already surfaced)
- `src/content/schema/tuning.ts` (Zod: new `getaway` fields)
- `src/platform/presets/build-config.ts` (map new fields)
- `presets/default/tuning.json` and `presets/test-malformed/tuning.json` (new fields)
- `src/engine/reduce.test.ts` / `src/engine/history.test.ts` (ditch + undo coverage)

**Do:**
- Add a pure `getawayBrief(heat, cfg): { targetCards: number; timerSeconds: number }` to `src/engine/getaway.ts`. It maps the Heat *at call time* to a card target and a countdown length, tuned so the difficulty curve tracks the model's odds bands (low Heat → small target / generous time ≈ the ~90% cool feel; Heat ≈ escape-signal (11) → ~50/50 pressure; near HMAX (20) → large target / short time ≈ the brutal 15–25% feel). Read all coefficients/bands from `cfg.getaway` — no hardcoded numbers. Keep it monotonic: `targetCards` non-decreasing in Heat, `timerSeconds` non-increasing in Heat. Pick a representable shape (e.g. linear interpolation between a `lowHeat` and `highHeat` anchor pair, or per-band thresholds) and round to integers.
- Add a `GETAWAY_DITCH` event to `RunEvent` and handle it in `reduce.ts`: it ticks Heat up by `cfg.getaway.ditchHeatCost` (clamped to HMAX like other Heat additions), pushing onto `history` so `UNDO_LAST` restores the prior state. It is a within-game penalty: it does **not** recompute the brief (difficulty is locked at call time per the design); its only downstream effect is raising the Heat used by `scoreRun`'s low-Heat style bonus at `RESOLVE_GETAWAY`.
- Extend `cfg.getaway` (config type + Zod schema + both preset `tuning.json` files + `build-config.ts` mapping) with the new tunables: the target/timer anchors or bands, `ditchHeatCost`, and a `buySecondsBonus` (seconds added per gear "buy seconds" spend, consumed by E6.2's UI). Keep the existing odds fields (`exponent`, `skillTerm`, `skillPivot`, `headcountTerm`, `clamp`) unchanged. Keep `.strict()` on the schema; ensure `test-malformed` stays parseable where the loader test expects it (mirror whatever shape the malformed fixture needs).

**Acceptance:**
- Unit tests: `getawayBrief` is monotonic across Heat 0→HMAX (target non-decreasing, timer non-increasing), returns integers, reads from preset (changing a tunable changes output), and lands within sane bands at Heat 0, ~11, and HMAX.
- `GETAWAY_DITCH` raises Heat by `ditchHeatCost` (clamped at HMAX) and is reverted by `UNDO_LAST`; reducer remains exhaustive/typed.
- `npm run check:full` green, including `sim:check` (the headless sim still uses the odds path — confirm the new fields don't perturb `getawayOdds`/`resolveGetawayOutcome`).

**Depends:** none (E1–E2 already merged)

## Task E6.2 — Console: live Getaway referee screen (replaces the stub)
**Files:**
- `src/console/screens/Getaway.tsx` (new — the live referee screen)
- `src/console/screens/Getaway.test.tsx` (new)
- `src/console/screens/index.tsx` (route `'getaway'` → `<Getaway />`; update exports)
- remove `src/console/screens/GetawayStub.tsx`; retarget `src/console/screens/GetawayStub.test.tsx` → `Getaway.test.tsx`
- (no engine changes — consumes E6.1)

**Do:**
- Build the live Getaway screen (`data-testid="screen-getaway"` to keep the phase-router assertion green). On mount it computes the brief once via `getawayBrief(state.heat, cfg)` and runs the round-the-circle Articulate game as a referee:
  - **Round-the-circle counter:** show the current clue-giver (cycles through the active crew in order) and a "cards cleared / target" readout. A **Cleared** action increments cards-cleared and advances to the next player.
  - **Countdown timer:** a real-time countdown from `timerSeconds`, held in local/store-ephemeral UI state (never in `RunState` — engine stays pure, no timers). Start/pause control for the GM.
  - **Ditch:** dispatches `GETAWAY_DITCH` (Heat tick, undoable) and moves on (advances the clue-giver without incrementing cards-cleared).
  - **Gear-spend affordances (GM referee, per the design reading above):** **Skip card** advances past the current card (clue-giver moves on, no clear); **Buy seconds** adds `cfg.getaway.buySecondsBonus` to the running countdown. These adjust UI timer/counter only.
  - **Resolution:** when cards-cleared reaches the target before time runs out → escape clean → dispatch `RESOLVE_GETAWAY { win: true }`; when the timer hits 0 → busted → dispatch `RESOLVE_GETAWAY { win: false }`. The engine then scores and moves to `result`.
- Keep GM override always available (golden rule #1, no dead-ends): explicit **Force win** / **Force bust** buttons (dispatch `RESOLVE_GETAWAY { win }`) and the global "undo last" continue to work from here.
- Add light, clearly-commented seams for climax narration and sound (intro / countdown / win / bust trigger points) — no E8/E9 wiring yet; just the integration points so E8/E9 can attach.

**Acceptance:**
- A full run reaches `getaway` and the screen runs the live game: clearing `target` cards before time-out dispatches `RESOLVE_GETAWAY { win: true }`; timer expiry dispatches `{ win: false }`; both land on `result` with the correct engine score/style bonus.
- Ditch dispatches `GETAWAY_DITCH` and is undoable; Skip card / Buy seconds adjust counter/timer; Force win / Force bust still resolve.
- Low Heat yields a generous brief (small target, long timer) and high Heat a brutal one (assert via the rendered target/timer for representative Heat values), matching the model bands.
- `npm run check:full` green; phase router test and existing console tests still pass.

**Depends:** E6.1

- `src/player-view/App.tsx` (route the new slice kind)
- `src/player-view/channel.ts` (re-export the new slice type)

**Do:**
- Add a Zod-validated `getaway` slice (mirroring the `defuse-rulebook` pattern) carrying **only player-safe data**: cards-cleared, target, seconds-remaining, current clue-giver name/index, and a `gameActive` flag. No GM-only state (no odds, no Heat internals, no score). Extra keys stripped on parse (parse-at-boundary, golden rules #6/#7).
- From the console Getaway screen, `publishSlice` the current display state as it changes, and `publishSlice({ kind: 'idle' })` when the game resolves or the screen unmounts (mirror Defuse's publish/idle lifecycle).
- Add a `GetawayDisplay` screen to the player-view and route it in `App.tsx` for the new slice kind. The player-view must not import from `@/console` (isolation enforced by the existing import-direction lint rule).

**Acceptance:**
- Slice schema round-trips player-safe fields and strips/rejects extras; player-view renders the Getaway display from a published slice and returns to idle on resolve.
- Player-view imports nothing from `@/console`; import-direction lint passes.
- `npm run check:full` green.

**Depends:** E6.2
