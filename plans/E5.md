# Plan — Epic E5: The ten mini-games

**Approach.** E4 already delivered the `MiniGame` contract, all six shared primitives (`Timer`, `CardSpread`, `Metronome`, `BoostButton`, `OutcomeJudge`, `DialReadout`), the generic `MinigameHost` launcher (resolves gameId → variant → dial → params → Component, with a stub fallback so nothing dead-ends), and **Safe-Crack** as the proven reference game. The engine-side data is also already in place: `presets/default/scaling.json` carries every game's `minCommit`, `variant` (incl. `crackTheTumblersSolo` and `assemblyLineNegotiated`), `excludedFromSolo`, and dial curves; `presets/default/content/roomTemplates.json` already lists all ten obstacles; and `resolveGameVariant` / `computeDial` / `obstacleCommitRange` already exist and are wired into `MinigameHost`. **So E5 is not new framework or new engine work — it is implementing the nine missing game modules (plus two solo/scaling variant modules), the two content banks they read, the player-view surface Defuse requires, and the per-game tests that prove the epic's acceptance gate.**

Each game task mirrors how Safe-Crack was delivered: a self-contained module folder under `src/minigames/games/<id>/` (`generate.ts` pure & RNG-only, `judge.ts` pure, `component.tsx` reusing the shared primitives, `index.ts` exporting the `MiniGame`), its own unit tests (generator reproducibility under seed+dial, `judge` over the three tier boundaries, each boost `apply` once-per-game), registration in `src/minigames/registry.ts`, and a `MinigameHost.<game>.test.tsx` integration test proving it launches inside the loop and `onResolve` feeds `RESOLVE_MINIGAME`. Tasks are ordered by the build-order groups in `docs/EPICS.md`/`docs/MINIGAMES.md §6` (shared-primitive reuse) and by the two genuine dependencies (the content-bank loader is introduced in the Categories task and reused by Inside Knowledge; everything else depends only on the merged E4 framework). The two games that touch shared infrastructure — Categories (content-bank loader) and Defuse (player-view surface + platform channel) — carry that infrastructure in their own task. Game ids are the camelCase keys already present in `scaling.json`.

Per-game **acceptance** is the epic gate applied to that game: it runs inside the loop replacing the stub; regenerates fresh each play from the seeded RNG (same seed+dial ⇒ same params); the `DialReadout` shifts with committed lane ratings; each boost surfaces only for a committed holder and fires once; `judge` suggests and the GM's `OutcomeJudge` tap is what feeds the engine; it respects its `minCommit`/variant; and `npm run check:full` stays green. The excluded-game and solo-variant invariants from the gate are proved by tests sited in the relevant tasks (Crack the Tumblers → solo variant resolves; Assembly Line & Defuse → `minCommit 2` floor means the generator/`obstacleCommitRange` never offers a sub-`minCommit` commit slot) and consolidated in the final sweep (E5.10).

The boost names, lanes, dial levers, `judge` tier shapes, and `minCommit`/variant rules for every game are fixed in `docs/MINIGAMES.md §6` (sourced from `docs/design/heist-content.md` and `heist-game-design.md`) — builders implement those, do not invent. Do **not** add or rename any `scaling.json` key, dial curve, or room-template obstacle — they already exist; a builder needing a *new* tunable must add it as preset data, never hardcode it.
