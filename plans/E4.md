# Plan — Epic E4: Mini-game framework + reference game

**Approach.** E4 builds the `MiniGame` plugin contract, the six shared primitives, the
console-side launcher that resolves a committed obstacle into a live game (variant →
dial → seeded `generate` → mount → GM-confirm → engine), and proves the whole shape
with one real game, **Safe-Crack**. The engine already ships everything the dial needs
(`computeDial`, `resolveGameVariant`, `obstacleCommitRange`, `rngFromState`, exported
from `@/engine`); the room loop already stores the committed crew and option on the
obstacle room and routes the `minigame` phase to `MinigameStub`. So E4 is almost
entirely a **minigames + console** job — no engine changes are required, and the only
content change is an explicit `safeCrack` dial-curve entry so the dial visibly shifts.

Two decisions worth stating up front, both derived from the docs (not invented):

1. **GameId convention.** The active preset keys games in camelCase (`safeCrack`,
   `crackTheTumblers`, …) — see `presets/default/content/roomTemplates.json` and
   `scaling.minCommit`. The registry and each `MiniGame.id` **must** use that same
   string, since the launcher looks games up by `option.gameId`. Use `safeCrack`
   (not the doc's prose spelling `safe-crack`) as the canonical id.
2. **Where mini-game RNG comes from (determinism, CLAUDE.md rule 4 / ARCHITECTURE §8).**
   `generate(rng, dial)` draws from the *run's* seeded stream, seeded read-only from
   the present `RunState.rngState` at minigame-entry. `CHOOSE_OPTION` does **not**
   advance `rngState`, so it is stable for the whole `minigame` phase; the launcher
   memoises params on `(rngState, resolvedGameId, dial)` so re-renders never reshuffle.
   Mini-game params are ephemeral UI state — they are **not** persisted in the
   `{seed, eventLog}` save and never feed the engine except via the GM-confirmed
   `RESOLVE_MINIGAME` outcome — so deriving them read-only keeps the engine pure and
   the run replays exactly (the replayed `rngState` re-derives identical params).
   The launcher must **not** mutate engine `rngState`.

Audio note: the platform audio engine is E9. For E4, `Timer`/`Metronome` use the Web
Audio API directly inside the minigames layer (a minimal oscillator beep); E9 refactors
them onto the shared engine. This matches the epic's "coordinate with E9" wording — E4
builds the primitive, E9 owns the engine.

Only Safe-Crack exists after E4; the other nine obstacle gameIds have no module yet
(they arrive in E5). The launcher must therefore **fall back to the existing
outcome-picker stub** for any gameId not in the registry — no dead-ends (CLAUDE.md
rule 1).

## Task E4.1 — MiniGame contract + registry
**Files:** `src/minigames/contract.ts` (new), `src/minigames/registry.ts` (new),
`src/minigames/index.ts` (replace placeholder; re-export contract + registry),
`src/minigames/contract.test.ts`, `src/minigames/registry.test.ts`
**Do:**
- Define the contract types exactly per `docs/MINIGAMES.md §1`:
  `Difficulty` (`{ level: number }`), `CommittedPlayer`
  (`{ id: PlayerId; name: string; stats: Record<Lane, number>; powerUps: Partial<Record<Lane, boolean>> }`
  — a player projection the launcher builds), `BoostHook`
  (`{ lane: Lane; label: string; apply(state, params): ChallengeState }`),
  `MiniGameProps<Params, ChallengeState>`
  (`{ params; dial; committed: CommittedPlayer[]; onResolve(o: Outcome): void }`),
  and `MiniGame<Params, ChallengeState>`
  (`{ id: GameId; lanes: Lane[]; generate(rng: Rng, dial: Difficulty): Params; Component; judge(state, params): Outcome; boosts: BoostHook[]; minCommit: number; soloVariantId?: GameId }`).
  Re-use `Lane`, `Outcome`, `PlayerId`, `GameId`, `Rng` from `@/engine` — do not redefine.
- `registry.ts`: a `Map<string, MiniGame<any, any>>` (or a record) populated from a
  `games: MiniGame[]` array (empty for now — populated in E4.4), plus
  `getGame(id: string): MiniGame | undefined` and a `hasGame(id)` helper. Keep the
  array as the single registration point so adding a game is one edit.
**Acceptance:** contract types compile under strict TS (no `any` leaking through the
public surface except the registry's intentional storage type); `getGame` returns
`undefined` for an unknown id and the registered game for a known one; `npm run check:full`
green. The import-direction lint stays green (minigames imports only `@/engine` types here).
**Depends:** none (E0–E3 merged).

## Task E4.2 — Shared primitives
**Files:** `src/minigames/primitives/{Timer,CardSpread,Metronome,BoostButton,OutcomeJudge,DialReadout}.tsx`
(new), `src/minigames/primitives/index.ts`, co-located `*.test.tsx` for each.
**Do:** build the six primitives from `docs/MINIGAMES.md §9`, each a small, reusable
component/hook:
- **`Timer`** — audible countdown with `seconds`, `running`, `onExpire`, optional
  `audible`; uses Web Audio for the tick/expiry beep (E9 will swap in the shared engine).
- **`CardSpread`** — `cards`, `layout: 'row'|'grid'`, `faceDown?`, `onTap?`; renders
  face-up/down cards, supports tap-to-select. (Not used by Safe-Crack, but an E4
  deliverable and exercised by its own test.)
- **`Metronome`** — `useMetronome({ bpm, audibleBeats })` precise Web Audio clock
  (not `setTimeout`); `onBeat`, `mute`. (E9 integrates with the audio bed.)
- **`BoostButton`** — renders **only** when a *committed* player holds the lane
  power-up for the hook's lane; fires `onFire(hook)` **once**, then disables; returns
  `null` when nobody committed holds the lane. (`docs/MINIGAMES.md §4`.)
- **`OutcomeJudge`** — clean/complication/botched control pre-set to `suggested`,
  the middle one tap away; `onConfirm(o)` is authoritative (GM confirms, app never
  overrides — `the-job-app-design.md §10.4`).
- **`DialReadout`** — GM-only display of the resolved `Difficulty`.
**Acceptance:** unit tests prove: `BoostButton` is `null` with no committed holder,
renders for a committed holder, and disables after one fire; `OutcomeJudge` defaults to
the suggested tier and emits the GM's chosen tier; `Timer` calls `onExpire` at zero;
`Metronome` emits beats and mutes on cue (test against its clock, not wall time);
`CardSpread` renders/taps. `npm run check:full` green; import-direction lint green.
**Depends:** E4.1.

## Task E4.3 — Mini-game launcher (dial wiring + registry resolution), replacing the stub
**Files:** `src/console/screens/MinigameHost.tsx` (new), `src/console/screens/index.tsx`
(route `minigame` phase to `MinigameHost`; keep `MinigameStub` as the fallback),
`src/console/screens/MinigameHost.test.tsx`. (`MinigameStub.tsx` stays for fallback.)
**Do:** implement the runtime launch from `docs/MINIGAMES.md §2`:
- Read `currentRoom` (must be obstacle), its `committedOptionId`/`committedBy`, and the
  committed option's `gameId` from the store.
- `getGame(gameId)`; if **absent**, render `MinigameStub` (graceful fallback — the nine
  E5 games aren't built yet; no dead-end).
- Resolve the effective module via `resolveGameVariant(gameId, commitSize, headcount, cfg)`
  and re-`getGame` the resolved id (falls back to stub if a variant module is missing).
- Build `CommittedPlayer[]` by projecting the committed `Player`s (id/name/stats/powerUps).
- Compute the dial: gather each committed player's ratings across the game's `lanes`,
  call `computeDial(ratings, resolvedGameId, headcount, cfg)`, wrap as `Difficulty`.
- Derive `rngFromState(present.rngState)` (read-only — **never** write `rngState` back),
  and `useMemo` the params on `(rngState, resolvedGameId, dial.level)` so the game
  generates once per entry and never reshuffles on re-render.
- Mount the game's `Component` with `MiniGameProps`; show `DialReadout` on the GM
  referee screen; on `onResolve(outcome)` dispatch `{ t: 'RESOLVE_MINIGAME', outcome }`.
**Acceptance:** with a **mock** `MiniGame` registered in the test, the host computes a
dial that *decreases* (eases) as committed stats rise, generates stable params across
re-renders (seeded), renders `DialReadout`, and forwards the component's `onResolve`
to a `RESOLVE_MINIGAME` dispatch; an obstacle whose gameId has no module falls back to
the outcome-picker stub. `npm run check:full` green.
**Depends:** E4.1, E4.2.

## Task E4.4 — Safe-Crack reference game module
**Files:** `src/minigames/games/safe-crack/{index.ts,generate.ts,judge.ts,component.tsx}`
(new) + `safe-crack.test.ts`; register it by adding to the `games` array in
`src/minigames/registry.ts`.
**Do:** implement Safe-Crack per `docs/MINIGAMES.md §6.7` with `id: 'safeCrack'`,
`lanes: ['tech','stealth']`, `minCommit: 1`:
- **`generate(rng, dial)`** (pure, RNG only): hidden code (digit count + value range),
  guess budget, timer. The dial drives the levers — easier (lower `dial.level`) ⇒
  fewer digits / more guesses / more time; Tech eases digit count, Stealth eases guess
  count (aggregation already folded into the dial value).
- **`ChallengeState`**: guesses-so-far with feedback ("two right, one in place"),
  guesses remaining, solved flag, per-lane boost-used flags.
- **`judge`** (app fully judges): **clean** = solved with guesses to spare;
  **complication** = solved on the last guess; **botched** = guesses exhausted unsolved.
- **`boosts`**: Tech → **Stethoscope** (reveal a digit's position); Stealth →
  **Patient Touch** (one extra guess) — each a pure `apply`, once per game.
- **`Component`**: reuses `Timer`, `BoostButton` (one per held lane), `OutcomeJudge`
  (pre-set to `judge`'s suggestion), `DialReadout`; lets the GM enter guesses, shows
  feedback, and confirms the outcome.
**Acceptance:** generator reproducibility — same seed + same dial ⇒ identical params,
and a higher dial yields more digits / fewer guesses (a tighter game); `judge` returns
the correct tier across its three boundaries; each boost `apply` is pure and effects the
state once; the game registers and `getGame('safeCrack')` returns it. `npm run check:full`
green; import-direction lint green (game imports `@/engine`, `@/content`, primitives only).
**Depends:** E4.1, E4.2.
