# Plan — Epic E10: Persistence, scoring history & polish

## Approach

E10 turns a functional-but-bare console into the finished product: the full
design-system visual pass on every surface, plus the personal-best leaderboard
that closes the loop on a finished run. Two facts from inspecting `main` shape
this plan:

1. **E0 shipped a *placeholder* token file, not the real one.** `src/console/theme/tokens.css`
   is a hand-rolled skeleton (`--color-bg`, `--font-sans: system-ui`, hardcoded
   hex) whose own header says *"Full cinematic polish is scheduled for E10."* The
   real design-system token file (`design-system/colors_and_type.css`, with
   `--bg-app`, `--font-display`, `--heat`, `--accent-tint`, the `t-*` type
   classes, the motion/radius tokens) was **never wired in**, fonts were **never
   self-hosted** (`public/fonts/` does not exist; the real token file still
   `@import`s Google Fonts CDN), and there is **no CDN-check sensor**. So the E0
   font/CDN stories land here, as the foundation of the visual pass — exactly
   what the E10 acceptance gate already demands ("no CDN requests at runtime;
   fonts and icons are bundled/self-hosted").
2. **No leaderboard exists.** Only a comment in `save.ts` references it. Run
   saves and settings persist under their own keys already (E3); the leaderboard
   is a new, parallel persistence concern.

Strategy for the visual pass: the designers already wrote the production-grade
stylesheet (`design-system/ui_kits/gm-console/kit.css`, `player-view/player.css`)
and reference JSX. We **adopt `kit.css` as the console stylesheet** and translate
the reference JSX class structure onto our existing typed components, preserving
every `data-testid` so the existing test suite and the QA script keep passing.
Foundation first (tokens, fonts, sensor, shared CSS, icon dep), then shared
primitives, then each surface. Leaderboard data is an independent slice built
early so the Setup and Result screens can consume it.

Verification reality: builders produce the *structural* match (correct tokens,
kit classes, kit DOM shape, Lucide icons, glow on the one live element); the
game-design/design reviewer and the QA agent confirm fidelity against the
`design-system/preview/*` specimens and assert no CDN requests at runtime. Every
task keeps `npm run check:full` green and honours the golden rules — engine
stays pure, all randomness via the seeded RNG, every tracked value GM-editable,
player-view leaks no GM-only state.

## Task E10.1 — Visual foundation: real tokens, self-hosted fonts, CDN sensor, icon bundle, kit stylesheets
**Files:**
- `src/console/theme/tokens.css` (replace placeholder with real token set)
- `src/console/theme/kit.css` (new — adopted from `design-system/ui_kits/gm-console/kit.css`)
- `src/console/index.css` (rebase global resets/body onto real tokens + kit)
- `src/console/theme/fonts.css` (new — local `@font-face`, no CDN) and `src/console/theme/index.ts` (export wiring as needed)
- `package.json` (add `lucide-react`; add the three font packages — see Do)
- `scripts/sensors/no-cdn.mjs` (new sensor) + wire into `check:full` in `package.json`
- `src/console/theme/tokens.test.ts` or similar (assert no CDN substring in the shipped token/font CSS)

**Do:**
- Replace the placeholder `tokens.css` with the contents of
  `design-system/colors_and_type.css`, **deleting the `@import url('https://fonts.googleapis.com/...')`
  line** (lines ~10–13 of the source). Do not alter token *values* — this file
  is the source of truth. The result must expose the real tokens the kit uses
  (`--bg-app`, `--fg`, `--font-display/-data/-body`, `--heat`, `--heat-glow`,
  `--accent`, `--accent-tint`, `--glow-heat/-accent/-data`, `--dur-fast/-base/-slow`,
  `--radius-*`, the `t-*` semantic type classes, etc.).
- **Self-host the three fonts offline (no CDN, bundled).** Use the Fontsource npm
  packages — `@fontsource/saira-condensed`, `@fontsource/jetbrains-mono`,
  `@fontsource/ibm-plex-sans` — which ship woff2 over npm (fully offline, no
  runtime CDN). Either import their CSS or author local `@font-face` blocks in
  `fonts.css` pointing at the package woff2 (covering the weights the design
  system uses: Saira 500/600/700/800, JetBrains 400/500/700/800, IBM Plex
  400/500/600/700 + italic 400). **Do not `curl` Google Fonts** — the box is
  offline at runtime and the CDN is banned. Import the font CSS at the app root
  (`src/console/index.css` or `main.tsx`).
- Adopt `design-system/ui_kits/gm-console/kit.css` as `src/console/theme/kit.css`
  verbatim (it already builds on the real tokens) and import it at the app root.
  Rebase `index.css` so the global reset/body use the real tokens (`--bg-app`,
  `--fg`, `--font-body`) instead of the deleted placeholder names.
- Add `lucide-react` as a dependency (bundled icon set — imported from the
  package, never CDN). No component changes required yet; this just makes the dep
  available to later tasks.
- Add `scripts/sensors/no-cdn.mjs`: run `npm run build` (or reuse an existing
  build artifact), then scan `dist/**/*.{css,js,html}` and assert **no** CDN
  hostname appears — at minimum `fonts.googleapis.com`, `fonts.gstatic.com`,
  `cdn.jsdelivr.net`, `unpkg.com`. Exit non-zero with a clear message if any is
  found. Wire it into `npm run check:full` after the import-direction sensor.

**Acceptance:**
- `tokens.css` contains the real design-system tokens and **no** `https://`
  font/CDN `@import`; `public/`-or-package woff2 are bundled; `--font-display`
  resolves to Saira Condensed, etc.
- `npm run build` succeeds; `node scripts/sensors/no-cdn.mjs` passes (no CDN
  hostname in `dist`) and is part of `check:full`.
- `lucide-react` installed; `kit.css` imported at root.
- `npm run check:full` green; the existing screen/HUD tests (which key off
  `data-testid`, not class names) still pass.

**Depends:** none (builds on merged E0–E9)

## Task E10.2 — Leaderboard & score history (persistence + store write-on-finish)
**Files:**
- `src/content/schema/leaderboard.ts` (new Zod schema + version constant)
- `src/content/schema/leaderboard.test.ts` (new)
- `src/platform/persistence/leaderboard.ts` (new — own storage key)
- `src/platform/persistence/leaderboard.test.ts` (new)
- `src/platform/index.ts` (export new API)
- `src/console/store/store.ts` + `src/console/store/store.test.ts` (write on run finish)

**Do:**
- Define a leaderboard entry schema (e.g. `{ seed, score, loot, heatAtGetaway,
  win, crewSize, ... }`) and an envelope `{ version, entries: Entry[] }`, Zod-
  validated, **schema-versioned** like `save`/`settings`. Parse at the boundary;
  malformed/stale → reset to empty, never throw at the table.
- `leaderboard.ts`: `readLeaderboard(storage)`, `appendScore(entry, storage)`
  (upsert), `clearLeaderboard(storage)`, plus a `topEntries(n)` / `personalBest()`
  helper. Use its **own** localStorage key (e.g. `the-job:leaderboard`), separate
  from the run save and settings, so clearing/finishing a run never wipes
  history. Swallow quota/security errors like the other persistence modules.
- Store: when a `dispatch` transition lands the session in the **result** phase
  with a final score, write the run to the leaderboard (write-through, beside the
  existing `writeSave`). **Dedupe by `runSeed`** (upsert keeping the best score)
  so undo/redo across the result boundary, or replaying, never double-counts.
  Hydrate/replay must not append (they don't go through `dispatch`). Expose a
  selector/getter the screens can read (e.g. current run's rank / new-best flag,
  and the top-N list).
- Reuse the engine's `scoreRun`/`getawayMultiplier` outputs already on the
  session — do **not** recompute scoring in the console (keep the engine the one
  source of truth).

**Acceptance:**
- A finished run appends exactly one leaderboard entry; finishing again (new
  seed) appends another; undoing out of result and back does not duplicate
  (dedupe-by-seed test).
- Leaderboard persists across reload (separate key); a corrupt/stale leaderboard
  resets cleanly without throwing.
- `personalBest`/`topEntries` return correctly ordered results.
- `npm run check:full` green; new schema validates in tests.

**Depends:** none (builds on merged E3 store/persistence; independent of E10.1)

### Epic acceptance gate (recap — verified by QA + design reviewer)
- A finished run appears on the leaderboard and persists across refresh/restart
  (E10.2, E10.6); mid-run resume still works — E3 regression check.
- Every GM-console screen + the player-view match the `design-system/preview/`
  specimens and are approved by the design reviewer (E10.3–E10.7).
- No CDN requests at runtime — fonts and Lucide icons bundled/self-hosted; the
  `no-cdn` sensor enforces it in `check:full` (E10.1).
- No major a11y/perf regressions; big hit targets, readable across a room.
