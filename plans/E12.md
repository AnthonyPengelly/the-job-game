# Plan — Epic E12: True-offline build (optional)

**Goal (from `docs/EPICS.md` E12 / `docs/design/the-job-app-design.md` §Epic 12):** a built artifact that double-clicks to play with **zero dev server** — runs fully from `file://`, no `localhost`, all features intact.

**Approach.** The docs settle the approach: "double-click to play, no `localhost`" mandates a static `file://`-friendly build, not a service worker (a SW still needs a server/secure-context to bootstrap, so it would not meet "no `localhost`"). Nothing offline-related exists on `main` today, so this is a fresh build.

The current `vite build` is incompatible with `file://` in three concrete ways, each owned by a task below:
1. **Absolute paths + external module scripts.** `dist/index.html` emits `base: '/'` absolute asset URLs and `<script type="module" crossorigin src="/assets/…">`. Under `file://` the absolute paths 404 and external module scripts are CORS-blocked. Fix: relative base (`'./'`) + single-file inlining (inline `type="module"` scripts run fine under `file://`) + inline fonts as data URIs. (E12.1)
2. **Audio fetched from `public/`.** Cues load via `fetch('sound/*.wav')`; `fetch()` of a `file://` resource is CORS-blocked, so audio silently degrades. Fix: bundle the sound buffers into the import graph so they inline as `data:` URIs and the browser audio provider resolves cue `src` → inlined URL (a `fetch('data:…')` works under `file://`). (E12.2)
3. **Player-view cross-window sync.** The optional player-view receives its slice over a `BroadcastChannel`, whose delivery across two `file://` documents (opaque/divergent origins) is unreliable. Fix: a `file://`-safe transport with graceful fallback, preserving player-view isolation. (E12.3)

A self-containment **sensor** then locks the result in against regression (E12.4), and a **QA + docs** pass verifies the double-click experience end-to-end and records the build/run procedure and any browser caveats (E12.5). The engine, content, and presets are untouched — this epic is entirely platform/build. `npm run dev` and `npm run check:full` must remain green throughout; persistence already swallows `SecurityError`, so localStorage simply degrades (never crashes) where a browser blocks it under `file://`.

---

## Task E12.1 — Relative-base, single-file offline build
**Files:** `vite.config.ts`; `package.json` (build/script wiring if a dedicated offline build is warranted); `index.html`, `player.html` (only if entry markup must change for inlining); possibly add a dev dependency (e.g. `vite-plugin-singlefile`) or a small post-build inliner under `scripts/`.
**Do:**
- Set `base: './'` so emitted asset references are relative, not root-absolute.
- Inline all JS and CSS into each HTML entry (`index.html` → `dist/index.html`, `player.html` → `dist/player.html`) so the built HTML carries no external `<script src>`/`<link href>` to `/assets/*`. Inline module scripts are permitted under `file://`; external module scripts are not — the inlining is what makes `file://` work.
- Raise `build.assetsInlineLimit` (or equivalent) so the self-hosted woff2 fonts inline as `data:` URIs inside the inlined CSS (no external font requests at runtime).
- Keep `npm run dev` (unbundled dev server) fully working and unchanged in behaviour; the offline shape applies to the production build only. Both entry points (`main`, `player`) must each build to a self-contained HTML.
**Acceptance:** after build, `dist/index.html` and `dist/player.html` each contain inlined script + style (no external `/assets/*` references, no `<script type="module" src=…>`); opening `dist/index.html` directly via `file://` loads the GM console with no console errors, no 404s, and no CORS/module-load failures (fonts render from inlined data URIs). `npm run check:full` stays green (including the existing `no-cdn` sensor) and `npm run dev` is unaffected.
**Depends:** none (builds on merged E0–E11).

## Task E12.2 — File://-safe audio (inline sound buffers)
**Files:** `src/console/audio/AudioProvider.tsx` (resolve cue `src` → bundled inlined URL via an injected `fetchBuffer`); a small asset-resolver module under `src/platform/audio/` or `src/console/audio/`; move/import the sound assets so Vite inlines them (e.g. relocate `public/sound/*.wav` into the import graph under `src/` and `import.meta.glob(..., { eager: true, as: 'url' })`, or per-file imports); `vite.config.ts` only if needed. Keep the sound manifest (`presets/default/content/sound.json`) as logical data — do **not** hardcode binary paths in the engine or schema.
**Do:**
- Make every cue buffer reachable under `file://` without a `fetch()` of a `file://` path. Resolve the preset manifest's logical `src` (e.g. `"sound/ambient-drone.wav"`) to a Vite-bundled, inlined `data:` URI, and feed the audio engine a `fetchBuffer` that fetches that resolved URI (`fetch('data:…')` succeeds under `file://`).
- Preserve presets-as-data: the manifest still lists logical cue srcs; the resolver maps logical → bundled asset. A missing mapping must fail loudly in dev (not silently at the table) while the engine's existing graceful-degradation on load failure remains intact.
- Keep `src/platform/audio/engine.ts` pure of bundler specifics (it already accepts an injected `fetchBuffer`); the resolution belongs in the console/platform wiring layer.
**Acceptance:** in the `file://` build the audio engine preloads cues successfully (`engine.loaded === true`, no fetch/decode errors in console) and soundboard/ambient/metronome cues play; existing audio tests (`src/platform/audio/*.test.ts`, `AudioProvider.wiring.test.tsx`) stay green; the offline build contains no runtime `fetch('sound/…')` of a file path (assets are inlined). `npm run check:full` green.
**Depends:** E12.1.

## Task E12.3 — File://-safe player-view sync
**Files:** `src/platform/channel/channel.ts` (transport); `src/player-view/channel.ts` / `src/player-view/App.tsx` (receive); minimal console wiring to obtain a window reference if a `postMessage` bridge is used; relevant tests.
**Do:**
- Ensure the optional player-view surface receives its slice when both the console and the player-view are loaded from `file://`. `BroadcastChannel` delivery across two `file://` documents is unreliable (opaque/divergent origins); provide an origin-agnostic fallback so the player-view still updates offline — e.g. publish the validated slice to a held player-window reference via `window.postMessage` (origin-agnostic), with the existing `BroadcastChannel` retained as the path for the served/dev case. If the builder confirms `BroadcastChannel` works under `file://` in the target browser, document that and keep the fallback as the safety net rather than removing it.
- **Preserve player-view isolation (golden rule 6):** only the existing Zod-validated `PlayerViewSlice` crosses the boundary — no GM-only state. The isolation test (`src/platform/channel/slice.test.ts`, `TuningPanel.test.tsx` import guard) must stay green.
- Degrade gracefully: if no transport is available, the console never blocks and the player-view simply shows no update (consistent with the existing try/catch no-op).
**Acceptance:** with console and player-view both opened from `file://`, a state change in the console (e.g. Getaway/Defuse slice) reaches the player-view; only player-safe fields cross the channel (isolation tests pass); console and dev mode behaviour unchanged; `npm run check:full` green.
**Depends:** E12.1.

## Task E12.4 — Offline self-containment sensor
**Files:** `scripts/sensors/offline-selfcontained.mjs` (new); `package.json` (`check:full` wires the new sensor); optionally a short note in `scripts/sensors/` conventions.
**Do:**
- Add a sensor (sibling to `no-cdn.mjs`) that builds from source and asserts each built HTML entry (`dist/index.html`, `dist/player.html`) is genuinely self-contained for `file://`: no external `<script src>`/`<link href>` to `/assets/*` or absolute `/`-rooted asset paths, no `<script type="module" src=…>`, no `http(s)://` asset references, and no leftover runtime `fetch('sound/…')` file-path references in the inlined JS. Exit non-zero with a clear message listing offenders.
- Wire it into `npm run check:full` so the offline property is a permanent regression gate. Reuse the build-from-source pattern of `no-cdn.mjs` so it can't false-pass on a stale `dist/`.
**Acceptance:** the sensor passes on the E12.1–E12.3 build and fails loudly if absolute paths, external module scripts, CDN URLs, or runtime sound-path fetches reappear; `npm run check:full` runs it and stays green.
**Depends:** E12.1, E12.2, E12.3.

## Task E12.5 — QA verification + offline build/run docs
**Files:** `docs/` (new short `docs/OFFLINE-BUILD.md` or a section in `docs/ARCHITECTURE.md`); `README`/build-script note if present; `HARNESS_CHANGELOG.md` if a harness lesson emerges. No application code beyond doc/wording.
**Do:**
- Verify the acceptance gate by hand/QA: build, then open `dist/index.html` from the filesystem (`file://`, no server) and play a complete run — Setup → run loop → a mini-game → Getaway → Result — confirming HUD, narration/teleprompter, audio (E12.2), and the player-view (E12.3) all function with zero network requests.
- Document the procedure: how to produce the offline artifact, how to launch it (double-click `dist/index.html`), and any browser caveat (notably that some browsers restrict `localStorage` under `file://` — the app degrades gracefully because saves swallow `SecurityError`, so resume/leaderboard may not persist in those browsers; recommend the target browser where it works).
- Note the player-view launch flow for the offline case (how the GM opens the player surface alongside the console under `file://`).
**Acceptance:** the built artifact runs fully offline from `file://` with no `localhost` server and all features intact (engine run, audio, narration, player-view); the offline build/run steps and browser caveats are documented; `npm run check:full` green.
**Depends:** E12.1, E12.2, E12.3, E12.4.
